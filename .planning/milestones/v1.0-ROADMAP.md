# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-02-16
**Phases:** 1-7
**Total Plans:** 18

## Overview

Cubano MVP delivers the core value proposition: a single, Pythonic query API that works identically across Snowflake and Databricks semantic views, with typed models, IDE autocomplete, and backend-agnostic code. This milestone spans from foundation (typed models) through production-ready packaging.

## Phases

### Phase 1: Model Foundation

**Goal**: Developers can define typed semantic view models with field references
**Depends on**: Nothing (first phase)
**Plans**: 1 plan

Plans:
- [x] 01-01-PLAN.md — Field descriptors, SemanticView base class, tests, and package metadata

**Details:**
Implemented typed models using Python metaclass with descriptor protocol. Field types (Metric, Dimension, Fact) validate identifiers at class creation time. SemanticView base class collects and freezes metadata, enabling immutable model objects. Foundation for all subsequent query and execution phases.

### Phase 2: Query Builder

**Goal**: Developers can construct immutable, type-safe queries with filters
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:
- [x] 02-01-PLAN.md — Q-object filter composition (TDD)
- [x] 02-02-PLAN.md — Immutable Query builder with method chaining (TDD)
- [x] 02-03-PLAN.md — OrderTerm with Field.asc()/desc() for descending ordering (TDD, gap closure)

**Details:**
Fluent query builder with immutable method chaining. Q-objects support AND/OR/NOT composition for complex filters. OrderTerm enables ascending/descending with NULL positioning control. All query operations return new Query instances, preserving immutability. Type-safe field references prevent runtime errors.

### Phase 3: SQL Generation & Mock Backend

**Goal**: Queries compile to SQL and execute against mock backend for testing
**Depends on**: Phase 2
**Plans**: 5 plans

Plans:
- [x] 03-01-PLAN.md — Engine ABC and Dialect architecture (Snowflake, Databricks, Mock)
- [x] 03-02-PLAN.md — SQLBuilder for composable SQL generation
- [x] 03-03-PLAN.md — MockEngine for testing without warehouse
- [x] 03-04-PLAN.md — Comprehensive tests for SQL generation and MockEngine
- [x] 03-05-PLAN.md — Gap closure: Fixture decoupling and MockEngine clarification

**Details:**
Engine ABC defines interface for all backends. Dialect pattern enables backend-specific SQL generation: MockDialect (Snowflake-style), SnowflakeDialect (AGG wrapper), DatabricksDialect (MEASURE wrapper). SQLBuilder composes SQL via string concatenation with placeholder support. MockEngine validates query structure without warehouse connection, enabling local testing with configurable fixtures. All phases build on proven SQL generation pattern.

### Phase 4: Execution & Results

**Goal**: Queries execute and return Row objects with attribute and dict access
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:
- [x] 04-01-PLAN.md — Row class with attribute and dict-style access (TDD)
- [x] 04-02-PLAN.md — Engine registry with register/get/unregister/reset (TDD)
- [x] 04-03-PLAN.md — Query.fetch(), Query.using(), MockEngine.execute(), public API integration

**Details:**
Row class supports dynamic field access via attribute and dict-style access, supporting queries with variable result shapes. Engine registry implements flat name→engine mapping with lazy resolution at fetch() time. Query.using() enables per-query engine selection. Public API exports Query, Row, and registry functions (register, get_engine, unregister). Integration tests verify end-to-end workflows.

### Phase 5: Snowflake Backend

**Goal**: Library connects to Snowflake and executes queries with AGG() syntax
**Depends on**: Phase 4
**Plans**: 2 plans

Plans:
- [x] 05-01-PLAN.md — SnowflakeEngine with lazy import, connection management, and error handling
- [x] 05-02-PLAN.md — Comprehensive unit tests for SnowflakeEngine with mocked connector

**Details:**
SnowflakeEngine lazily imports snowflake.connector only when instantiated, preventing ImportError for users without driver installed. Connection lifecycle managed via context managers. Error translation from Snowflake exceptions to RuntimeError provides consistent error contract. Test suite uses sys.modules patching for driver simulation without warehouse credentials.

### Phase 6: Databricks Backend

**Goal**: Library connects to Databricks and executes queries with MEASURE() syntax
**Depends on**: Phase 5
**Plans**: 2 plans

Plans:

- [x] 06-02-PLAN.md — Comprehensive unit tests for DatabricksEngine with mocked connector

DatabricksEngine implements same lazy import and connection management patterns as SnowflakeEngine, enabling coexistence of multiple backends. ## Milestone Summary

**Key Decisions:**
- Metaclass over decorator — cleaner ORM-style syntax for model definition
- Field refs only (no strings) — type safety and IDE autocomplete
- Mock backend first — validate design before production backends
- Dialect pattern — enables Snowflake/Databricks coexistence
- Engine registry lazy resolution — models importable before engines configured
- Standard SQL over SEMANTIC_VIEW() — consistent across backends
- Custom Row class — supports dynamic field access patterns

**Issues Resolved:**
- Field validation catches SQL injection patterns and reserved keywords at class definition
- Q-object composition supports arbitrary nesting with AND/OR/NOT
- SQL generation handles both aggregate (AGG/MEASURE) and dimension/fact selections
- MockEngine enables testing without warehouse or credentials
- Engine registry prevents ImportError for users without specific backend drivers
- Row objects support both attribute and dict-style access for ergonomic result handling
- Public API properly typed with py.typed marker for IDE integration

**Technical Debt:**
None incurred. All phases executed according to plan with minimal non-blocking test updates.
