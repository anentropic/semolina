### Pattern 1: v0.3 Connection Registration (replaces Engine registration)

**What:** Two supported patterns for connecting to a warehouse.

**Pattern A -- TOML config (recommended):**
```python
from semolina import register, pool_from_config

pool, dialect = pool_from_config()  # reads .semolina.toml
register("default", pool, dialect=dialect)
```

**Pattern B -- Manual pool construction:**
```python
from adbc_poolhouse import SnowflakeConfig, create_pool
from semolina import register

config = SnowflakeConfig(account="...", user="...", password="...")
pool = create_pool(config)
register("default", pool, dialect="snowflake")
```

**When to use:** Pattern A for most users (simple config file). Pattern B when credentials come from vault/secrets manager or need programmatic construction.

### Pattern 2: SemolinaCursor Result Access (replaces direct iteration)

**What:** `.execute()` returns `SemolinaCursor`, not a list of Rows.

**Primary pattern (Row convenience):**
```python
cursor = Sales.query().metrics(Sales.revenue).dimensions(Sales.country).execute()
rows = cursor.fetchall_rows()  # list[Row]
for row in rows:
    print(row.country, row.revenue)
```

**Arrow/raw pattern (DBAPI passthrough):**
```python
cursor = Sales.query().metrics(Sales.revenue).dimensions(Sales.country).execute()
raw = cursor.fetchall()  # list[tuple]
```

**Context manager pattern:**
```python
with Sales.query().metrics(Sales.revenue).execute() as cursor:
    rows = cursor.fetchall_rows()
```

### Pattern 3: MockPool for tutorials (replaces MockEngine)

**What:** MockPool provides the same pool.connect() interface.
```python
from semolina import MockPool, register

pool = MockPool()
pool.load("sales", [{"revenue": 1000, "country": "US"}])
register("default", pool, dialect="mock")
```

### Pattern 4: Query Shorthand (new in v0.3)

**What:** `query()` accepts keyword args as shorthand.
```python
# Shorthand
cursor = Sales.query(
    metrics=[Sales.revenue],
    dimensions=[Sales.country],
).execute()

# Equivalent fluent chain
cursor = (
    Sales.query()
    .metrics(Sales.revenue)
    .dimensions(Sales.country)
    .execute()
)
```

### Anti-Patterns to Avoid
- **Mentioning Engine API:** Write as if it never existed. No "previously" or "instead of".
- **Mentioning SnowflakeCredentials/DatabricksCredentials:** These are replaced by `.semolina.toml` + `pool_from_config()` and manual pool construction.
- **Iterating directly over `execute()` result:** The cursor is not iterable. Must call `.fetchall_rows()`, `.fetchone_row()`, or `.fetchmany_rows()`.
- **Using `for row in results:` pattern from v0.2:** Must explicitly show cursor fetch methods.

## Common Pitfalls

### Pitfall 1: Mixing old and new API patterns in examples
**What goes wrong:** Accidentally showing `SnowflakeEngine`, `MockEngine`, `engine.execute()`, or direct result iteration.
**Why it happens:** Existing docs are v0.2 and easy to miss during partial updates.
**How to avoid:** Every doc file must be reviewed against the v0.3 API checklist before the plan is marked complete. Search for "Engine", "engine", "SnowflakeCredentials", "DatabricksCredentials", "for row in results", "for row in query.execute()".
**Warning signs:** `uv run mkdocs build --strict` may not catch semantic errors (only link/reference errors).

### Pitfall 2: Forgetting SemolinaCursor is not iterable
**What goes wrong:** Writing `for row in cursor:` which does not work -- SemolinaCursor has no `__iter__`.
**Why it happens:** v0.2 `execute()` returned a list that was directly iterable.
**How to avoid:** Always show explicit fetch method: `cursor.fetchall_rows()`, `cursor.fetchone_row()`, `cursor.fetchmany_rows()`.

### Pitfall 3: Not updating cross-references and "See also" links
**What goes wrong:** Links pointing to removed concepts (e.g. credential loading pages that no longer exist).
**Why it happens:** Pages reference each other extensively via "See also" sections.
**How to avoid:** After all pages are updated, run `uv run mkdocs build --strict` which catches broken links.

### Pitfall 4: MockPool requires dialect="mock" in register()
**What goes wrong:** Showing `register("default", pool)` without `dialect=` for MockPool.
**Why it happens:** v0.2 `register()` did not require `dialect`.
**How to avoid:** All register() calls in docs MUST include `dialect=`. For MockPool use `dialect="mock"`.

### Pitfall 5: Installation extras have changed for Snowflake
**What goes wrong:** Docs still say `semolina[snowflake]` installs `snowflake-connector-python`.
**Why it happens:** v0.3 changed the snowflake extra to `adbc-poolhouse[snowflake]`.
**How to avoid:** Update `installation.md` and backend how-to pages to reflect the actual extra dependency.

### Pitfall 6: Forgetting the humanizer pass
**What goes wrong:** AI-sounding language ("leverage", "seamlessly", "robust") in documentation.
**Why it happens:** Natural tendency when drafting technical content.
**How to avoid:** The `semolina-docs-author` skill includes a mandatory humanizer pass for all new prose and major rewrites.

## Code Examples

### Complete Tutorial Example (v0.3 with MockPool)
```python
from semolina import (
    SemanticView,
    Metric,
    Dimension,
    MockPool,
    register,
)


class Sales(SemanticView, view="sales"):
    revenue = Metric()
    cost = Metric()
    country = Dimension()
    region = Dimension()


pool = MockPool()
pool.load(
    "sales",
    [
        {"revenue": 1000, "cost": 100, "country": "US", "region": "West"},
        {"revenue": 2000, "cost": 200, "country": "CA", "region": "West"},
        {"revenue": 500, "cost": 50, "country": "US", "region": "East"},
    ],
)
register("default", pool, dialect="mock")

cursor = (
    Sales.query()
    .metrics(Sales.revenue)
    .dimensions(Sales.country)
    .execute()
)

for row in cursor.fetchall_rows():
    print(row.country, row.revenue)
```

### SemolinaCursor Fetch Methods
```python
cursor = Sales.query(
    metrics=[Sales.revenue],
    dimensions=[Sales.country],
).execute()

# Row objects (primary pattern)
rows = cursor.fetchall_rows()        # list[Row]
row = cursor.fetchone_row()          # Row | None
batch = cursor.fetchmany_rows(10)    # list[Row]

# Raw DBAPI tuples
raw = cursor.fetchall()              # list[tuple]
raw_one = cursor.fetchone()          # tuple | None
raw_batch = cursor.fetchmany(10)     # list[tuple]

# Context manager (closes cursor + connection on exit)
with Sales.query(metrics=[Sales.revenue]).execute() as cursor:
    rows = cursor.fetchall_rows()
```

   - Recommendation: Codegen is a CLI tool separate from the query API. Keep credential env var references for codegen since it uses engines internally. Add a note that codegen credential setup is independent from query-time pool configuration.

2. **Warehouse testing how-to**
   - What we know: The how-to references `backend_engine` fixtures and `MockEngine`.
   - What's unclear: Have the test fixtures in `tests/integration/conftest.py` been updated to use MockPool?
   - Recommendation: Check the actual test fixtures during implementation. If they still use MockEngine, the warehouse testing how-to may only need terminology updates rather than code changes.

3. **adbc-poolhouse config field names**
   - What we know: TOML fields must match adbc-poolhouse config class fields exactly (per .semolina.toml.example).
   - What's unclear: The exact field names for DatabricksConfig (`host` vs `server_hostname`, `token` vs `access_token`).
   - Recommendation: During implementation, verify field names from the actual adbc-poolhouse config classes. The test file shows `host` for Databricks, not `server_hostname`.
