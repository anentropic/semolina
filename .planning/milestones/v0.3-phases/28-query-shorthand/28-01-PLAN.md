@dataclass(frozen=True, repr=False)
class _Query:
    _metrics: tuple[Metric[Any], ...] = field(default_factory=tuple)
    _dimensions: tuple[Dimension[Any] | Fact[Any], ...] = field(default_factory=tuple)
    _model: type | None = field(default=None, init=False, repr=False)

    def metrics(self, *fields: Any) -> _Query:
        # Validates isinstance(f, Metric), validates field ownership, concatenates tuples
        return self._replace(_metrics=self._metrics + fields)

    def dimensions(self, *fields: Any) -> _Query:
        # Validates isinstance(f, (Dimension, Fact)), validates field ownership, concatenates tuples
        return self._replace(_dimensions=self._dimensions + fields)
```

From src/semolina/models.py:
```python
class SemanticView(metaclass=SemanticViewMeta):
    @classmethod
    def query(cls, using: str | None = None) -> "_Query":
        from .query import _Query as QueryImpl
        q = QueryImpl(_using=using)
        object.__setattr__(q, "_model", cls)
        return q
```

CRITICAL ordering constraint (from RESEARCH.md Pitfall 1):
`_model` MUST be set via `object.__setattr__` BEFORE chaining `.metrics()` / `.dimensions()`,
because those methods validate field ownership via `self._model`.

Implementation approach (from RESEARCH.md recommendation):
Use the chaining approach -- call existing `.metrics()` and `.dimensions()` methods after setting `_model`.
This reuses existing validation code with zero duplication. Treat `metrics=[]` and `metrics=None` as no-op.

Design decision (from RESEARCH.md):
Make `metrics`, `dimensions`, and `using` all keyword-only (after `*`). All documented examples
already use keyword syntax. `using` was positional-or-keyword but is being made keyword-only.

to:
```python
@classmethod
def query(
    cls,
    *,
    metrics: Sequence[Metric[Any]] | None = None,
    dimensions: Sequence[Dimension[Any] | Fact[Any]] | None = None,
    using: str | None = None,
) -> "_Query":
```

3. Update the method body to chain builder methods AFTER setting `_model`:
```python
from .query import _Query as QueryImpl

q = QueryImpl(_using=using)
object.__setattr__(q, "_model", cls)

if metrics:
    q = q.metrics(*metrics)
if dimensions:
    q = q.dimensions(*dimensions)

return q
```

Key points:
- `if metrics:` is truthy check -- `None` and `[]` both evaluate to False, so they're no-ops (avoids Pitfall 3 from research: empty list would raise ValueError from .metrics())
- `_model` is set BEFORE chaining (Pitfall 1 from research: field ownership validation needs _model)
- `*` before params makes all args keyword-only (Pitfall 2: `using` becomes keyword-only too)
- No validation duplication -- `.metrics()` and `.dimensions()` already validate types and ownership

4. Update the docstring to document the new parameters:
```python
"""
Create a query for this semantic view.

Entry point for model-centric query construction. Accepts optional
shorthand keyword arguments for metrics and dimensions, or returns
a bare _Query for fluent method chaining.

Shorthand:
    Sales.query(
        metrics=[Sales.revenue],
        dimensions=[Sales.region],
    ).execute()

Fluent:
    Sales.query().metrics(Sales.revenue).dimensions(
        Sales.region
    ).execute()

Both styles can be combined -- builder methods are additive:
    Sales.query(metrics=[Sales.revenue]).metrics(Sales.cost)
    # selects both revenue and cost

Args:
    metrics: Optional list of Metric fields to select
    dimensions: Optional list of Dimension or Fact fields to group by
    using: Optional pool/engine name (defaults to 'default')

Returns:
    _Query instance bound to this model's view

Example:
    ```python
    cursor = Sales.query(
        metrics=[Sales.revenue],
        dimensions=[Sales.region],
    ).execute()
    ```
"""
```
