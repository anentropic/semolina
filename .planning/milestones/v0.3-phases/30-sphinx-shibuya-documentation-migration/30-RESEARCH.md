# Phase 30: Sphinx Shibuya Documentation Migration - Research

**Researched:** 2026-04-08
**Domain:** Documentation tooling migration (MkDocs Material -> Sphinx + shibuya)
**Confidence:** HIGH

## Summary

This phase replaces the MkDocs Material documentation stack with Sphinx + shibuya theme, converting all 16 content pages from Markdown to RST and replacing mkdocstrings with sphinx-autoapi for API reference generation. The migration is a format conversion with no content changes.

The current docs use 57 tab blocks (pymdownx.tabbed `=== "Name"` syntax), 10 admonitions (`!!! type`), 2 grid card sections, 1 Material button, and 17 mkdocstrings directives. All of these have direct RST equivalents via sphinx-design (tab-set, grid, card directives), native RST admonitions, and sphinx-autoapi. The shibuya theme has first-class support for sphinx-design components including an exclusive `surface` class for grid cards and an `outline` class for tab-sets.

**Primary recommendation:** Place `conf.py` in `docs/src/` (the source directory), keep all RST files in the same `docs/src/` tree, and use `sphinx-build -W docs/src docs/_build` as the build command. This preserves the existing directory structure and minimizes CI changes.

## Architecture Patterns

### Recommended Project Structure

```
docs/
  src/                     # Sphinx source directory (conf.py lives here)
    conf.py                # Sphinx configuration
    index.rst              # Homepage
    tutorials/
      index.rst
      installation.rst
      first-query.rst
    how-to/
      index.rst
      models.rst
      queries.rst
      filtering.rst
      ordering.rst
      codegen.rst
      warehouse-testing.rst
      backends/
        overview.rst
        snowflake.rst
        databricks.rst
    explanation/
      index.rst
      semantic-views.rst
    _static/               # Static assets (if needed)
  _build/                  # Build output (gitignored)
  scripts/                 # DELETE: gen_ref_pages.py no longer needed
```

**Key decisions:**
- `conf.py` lives in `docs/src/` (the source directory) -- Sphinx expects conf.py in the source root [VERIFIED: sphinx-doc.org/en/master/man/sphinx-build.html]
- No separate `-c` flag needed -- sphinx-build uses `docs/src` as both source and config directory
- `docs/_build/` replaces MkDocs `site/` output directory
- Reference pages are NOT manually created -- sphinx-autoapi generates them into `autoapi/` subdirectory automatically [VERIFIED: sphinx-autoapi.readthedocs.io]

### Pattern 2: Tab-set with sync-group (replaces pymdownx.tabbed)

**What:** Synchronized Snowflake/Databricks tab blocks
**When to use:** Every `=== "Snowflake"` / `=== "Databricks"` tab pair

```rst
.. tab-set::
   :sync-group: warehouse

   .. tab-item:: Snowflake
      :sync: snowflake

      .. code-block:: sql

         SELECT AGG("revenue"), "country"
         FROM "sales"
         GROUP BY ALL

   .. tab-item:: Databricks
      :sync: databricks

      .. code-block:: sql

         SELECT MEASURE(`revenue`), `country`
         FROM `sales`
         GROUP BY ALL
```

[VERIFIED: sphinx-design.readthedocs.io/en/latest/tabs.html -- sync-group and sync options confirmed]

**Key rules:**
- `:sync-group: warehouse` is the SAME value on every tab-set across all pages -- this makes tabs sync globally
- `:sync: snowflake` and `:sync: databricks` are the sync keys per tab-item
- For non-warehouse tabs (e.g., pip/uv in installation.rst), use a DIFFERENT sync-group (e.g., `:sync-group: installer`)
- Content must be indented under each `.. tab-item::` directive

### Pattern 4: Admonitions (replaces pymdownx !!! syntax)

**What:** Note, tip, warning boxes
**When to use:** Every `!!! type "Title"` block

MkDocs:
```markdown
!!! tip "Inspect SQL before executing"
    Use `.to_sql()` during development...
```

RST:
```rst
.. tip:: Inspect SQL before executing

   Use ``.to_sql()`` during development...
```

Or without a title:
```rst
.. tip::

   Use ``.to_sql()`` during development...
```

[VERIFIED: docutils.sourceforge.io/docs/ref/rst/directives.html -- standard RST admonition syntax]

**Admonition type mapping (all 1:1):**
- `!!! tip` -> `.. tip::`
- `!!! note` -> `.. note::`
- `!!! warning` -> `.. warning::`

### Pattern 5: Cross-references (replaces Markdown links)

**What:** Links between docs pages
**When to use:** All internal links

MkDocs: `[Building queries](queries.md)` or `[Filtering](../how-to/filtering.md)`

RST: `:doc:`queries`` or `:doc:`../how-to/filtering``

For links with custom text:
RST: `:doc:`Building queries <queries>``

External links:
RST: `` `Link text <https://example.com>`_ ``

[VERIFIED: sphinx-doc.org/en/master/usage/restructuredtext/basics.html]

### Anti-Patterns to Avoid

- **Do NOT use MyST Markdown:** The decision is RST, not MyST. Do not add myst-parser.
- **Do NOT manually create reference/*.rst files:** sphinx-autoapi generates them. Creating manual files causes duplicate object warnings.
- **Do NOT use `.. code::` for code blocks:** Use `.. code-block:: python` (with language) for proper highlighting.
- **Do NOT forget the blank line after directives:** RST requires a blank line between the directive declaration and its content. Missing it causes silent failures.
- **Do NOT mix tab sync-group names:** All Snowflake/Databricks tabs MUST use the same sync-group value for cross-page sync to work.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| API reference pages | Manual .rst files per module | sphinx-autoapi `autoapi_dirs` | autoapi handles discovery, filtering, and page generation automatically |
| Tab synchronization | Custom JavaScript | sphinx-design `sync-group` + `sync` | Built-in sync across all tab-sets on the page, plus URL query parameter support |
| Copy buttons on code | Custom JavaScript | sphinx-copybutton extension | Handles prompt stripping, accessibility, all code blocks |
| Google docstring parsing | Manual RST formatting in docstrings | sphinx.ext.napoleon | Converts Google-style to RST transparently |
| Dark mode toggle | Custom CSS/JS | shibuya built-in `color_mode: "auto"` | Follows OS preference by default, user-toggleable |

**Key insight:** sphinx-autoapi eliminates the entire `gen_ref_pages.py` script and the `reference/` subdirectory of manually-generated .md files. The 17 current mkdocstrings directives (`::: module.name`) and 17 auto-generated reference .md files are all replaced by autoapi configuration alone.

## Common Pitfalls

### Pitfall 1: RST indentation sensitivity

**What goes wrong:** Content inside directives doesn't render or causes build warnings.
**Why it happens:** RST uses whitespace-significant indentation (3 spaces is conventional). Content must be indented relative to the directive, with a blank line after the directive line.
**How to avoid:** Always use 3-space indentation inside directives. Always leave a blank line between `.. directive::` and content. Use `sphinx-build -W` (warnings as errors) to catch issues early.
**Warning signs:** `WARNING: Unexpected indentation` or content appearing as plain text outside the directive box.

### Pitfall 2: autoapi_dirs relative path

**What goes wrong:** sphinx-autoapi finds no modules, generates empty reference.
**Why it happens:** `autoapi_dirs` paths are relative to the directory containing `conf.py`. With conf.py in `docs/src/`, the path to `src/semolina/` is `../../src/semolina`.
**How to avoid:** Test with `sphinx-build -W docs/src docs/_build` and verify the `reference/` section appears in the toctree.
**Warning signs:** No `autoapi/` directory in build output, or "toctree contains reference to nonexisting document" warnings.

### Pitfall 3: Inline code in RST uses double backticks

**What goes wrong:** Single-backtick code renders as emphasized text (italic) instead of inline code.
**Why it happens:** RST uses single backticks for interpreted text roles (default role is usually emphasis), while Markdown uses them for inline code.
**How to avoid:** Convert all `` `code` `` (Markdown single backtick) to ``` ``code`` ``` (RST double backtick). This is a bulk find-replace across all files.
**Warning signs:** Code fragments appearing in italics in the rendered output.

### Pitfall 4: Tab-set content must be a proper directive body

**What goes wrong:** Tab content appears outside the tab, or tabs show empty.
**Why it happens:** sphinx-design tab-set/tab-item are nested directives. Each level needs proper indentation and blank lines.
**How to avoid:** Follow the exact indentation pattern from the Code Examples section. The tab content is indented under `.. tab-item::`, and any sub-directives (like `.. code-block::`) are further indented.
**Warning signs:** Content appearing after the tab-set instead of inside tabs.

### Pitfall 5: Admonition titles in RST

**What goes wrong:** Admonition renders without its custom title.
**Why it happens:** In MkDocs, `!!! tip "Custom Title"` puts the title inline. In RST, `.. tip:: Custom Title` also works for single-line titles, but multi-line content requires the title on the directive line and content indented below with a blank line separator.
**How to avoid:** For titled admonitions, use `.. tip:: Title text` followed by blank line and indented body. For untitled, use `.. tip::` followed by blank line and indented body.
**Warning signs:** Title text appearing as body content, or warning about unexpected content.

### Pitfall 6: Definition lists in RST differ from Markdown

**What goes wrong:** `**"Snapshot mismatch" after a refactor**\n:   Re-record with...` (used in warehouse-testing.md troubleshooting) doesn't render.
**Why it happens:** RST definition lists use a different syntax from Markdown.
**How to avoid:** Convert to RST definition list syntax:
```rst
"Snapshot mismatch" after a refactor
   Re-record with ``--snapshot-update`` and review the diff.
```
**Warning signs:** Definition term and body appearing on the same line or not styled as a list.

### Pitfall 7: GitHub Pages artifact path changes

**What goes wrong:** Deployment fails because the artifact path points to `site/` (MkDocs output) instead of `docs/_build/` (Sphinx output).
**Why it happens:** The existing docs.yml workflow uses `path: site/` for `upload-pages-artifact`.
**How to avoid:** Update `docs.yml` to use `path: docs/_build/` and change the build command from `uv run mkdocs build --strict` to `uv run sphinx-build -W docs/src docs/_build`.
**Warning signs:** GitHub Actions deployment step fails with "path not found".

### Pitfall 8: Tables in RST

**What goes wrong:** RST tables are verbose and error-prone compared to Markdown pipe tables.
**Why it happens:** RST has two table syntaxes: grid tables (complex) and simple tables (limited). Neither matches Markdown pipe tables.
**How to avoid:** Use the `.. list-table::` directive or the `csv-table` directive for all tables. These are more maintainable than grid tables:
```rst
.. list-table::
   :header-rows: 1

   * - Field
     - Use for
     - Accepted by
   * - ``Metric``
     - Aggregated measures
     - ``.metrics()``
```
**Warning signs:** Misaligned grid table borders causing build errors.

How to build queries
====================

Build queries using Semolina's fluent, immutable API. Chain ``.metrics()``,
``.dimensions()``, ``.where()``, ``.order_by()``, and ``.limit()`` to shape
your query, then call ``.execute()`` to get results.

This guide uses the ``Sales`` model from :doc:`../tutorials/first-query`:

.. code-block:: python

   from semolina import SemanticView, Metric, Dimension


   class Sales(SemanticView, view="sales"):
       revenue = Metric()
       cost = Metric()
       country = Dimension()
       region = Dimension()

Select metrics
--------------

Use ``.metrics()`` to choose which aggregated measures to include:

.. code-block:: python

   query = Sales.query().metrics(Sales.revenue)

.. tab-set::
   :sync-group: warehouse

   .. tab-item:: Snowflake
      :sync: snowflake

      .. code-block:: sql

         SELECT AGG("revenue")
         FROM "sales"

   .. tab-item:: Databricks
      :sync: databricks

      .. code-block:: sql

         SELECT MEASURE(`revenue`)
         FROM `sales`

See also
--------

- :doc:`filtering` -- field operators and boolean composition
- :doc:`ordering` -- sort results and control row counts
