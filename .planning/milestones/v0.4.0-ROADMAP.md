# Milestone v0.4.0: DuckDB Backend & Arrow Output

**Status:** ✅ SHIPPED 2026-05-07
**Phases:** 33–38
**Total Plans:** 12

## Overview

Milestone v0.4.0 adds DuckDB as a first-class backend alongside Snowflake and Databricks, exposes Arrow-native cursor output via `fetch_arrow_table()`, replaces MockPool with real DuckDB for testing, ships reverse codegen for DuckDB semantic views, and updates documentation to cover all three backends.

## Phases

**Goal**: Users can generate correct DuckDB `semantic_view()` SQL and install the DuckDB backend as a pip extra
**Depends on**: Nothing (first phase of v0.4.0)
**Plans**: 2 plans

Plans:

- [x] 33-01: DuckDB dialect, SQL builder, factory dispatch, and pip extra
- [x] 33-02: TOML config DuckDB entry and query execution wiring

**Details:**
Added `Dialect.DUCKDB` with `DuckDBDialect` (qmark paramstyle) registered at `engines/sql.py`. `DuckDBSQLBuilder` overrides `build_select_with_params()` to emit `SELECT ... Shipped `semolina[duckdb]` extra pinning `duckdb>=1.5.0` and `pyarrow>=17.0.0`.

### Phase 34: Arrow Output

**Goal**: Users can retrieve query results as a `pyarrow.Table` directly from `SemolinaCursor` via `fetch_arrow_table()`
**Depends on**: Nothing (independent of DuckDB SQL layer)
**Plans**: 1 plan

Plans:

- [x] 34-01: TDD: `fetch_arrow_table()` method + DuckDB ADBC tests

**Details:**
Added `fetch_arrow_table()` to `SemolinaCursor` delegating to the underlying ADBC cursor. The method returns the full result set as a single `pyarrow.Table`, suitable for zero-copy conversion to Pandas (`.to_pandas()`) or Polars (`pl.from_arrow(...)`). Since `pyarrow` is a transitive dependency of every backend ADBC driver, no separate guard is required and ARROW-01/03 are satisfied by design.

**Note:** Planning artifacts for this phase were deleted in commit `2933df2` on 2026-04-26. The audit verified delivery via direct code spot-checks.

**Naming note:** The original requirement text named the API `to_arrow()`; the shipped public API is `fetch_arrow_table()`, matching ADBC convention. Documentation reflects the shipped name.

- [x] 36-01: `DuckDBEngine` introspection + `duckdb_type_to_python` type mapping
- [x] 36-02: CLI `--backend duckdb` wiring with `--database` option

**Details:**
`DuckDBEngine` performs two-step introspection (`DESCRIBE SEMANTIC VIEW` + `DESCRIBE SELECT`) using the native `duckdb` driver. `duckdb_type_to_python()` maps 21 DuckDB SQL type entries to Python annotations, with an explicit `f"TODO: {sql_type}"` placeholder for unmappable types (intentional, tested behaviour). ## Milestone Summary

**Decimal Phases:** None — all phases ran in numeric order 33 → 38.

**Key Decisions:**

- **DuckDBSQLBuilder overrides `build_select_with_params()` entirely.** The DuckDB `semantic_view()` is a table function, not a view; the standard SELECT-from-view path doesn't apply. Outcome: ✓ Good — clean separation, dialect-specific behaviour stays in the builder.
- **DuckDB ADBC paramstyle is qmark, not pyformat.** Outcome: ✓ Good — caught early via the dialect registry, no leaked Snowflake/Databricks placeholder assumptions.
- **`fetch_arrow_table()` not `to_arrow()`.** Aligns with ADBC's existing fetch_arrow_table convention. Outcome: ✓ Good — consistent with the underlying transport. Requirement text (`ARROW-02`) named the older `to_arrow()` form; docs and audit reconciled to the shipped name.
- **MockPool removed entirely, no deprecation path.** v0.3 already shipped MockPool as `Out of Scope: clean break`. DuckDB in-memory pools are the replacement. Outcome: ✓ Good — testing surface is now real ADBC, no mock/prod divergence.
- **DuckDB extension auto-loads via `event.listen(pool, "connect", ...)`.** SQLAlchemy event listener installed at pool construction. Outcome: ✓ Good — zero per-query overhead, transparent to user code.
- **Two-step introspection (`DESCRIBE SEMANTIC VIEW` + `DESCRIBE SELECT`) using native `duckdb` driver, not ADBC.** Codegen is offline tooling, not a runtime path; native driver is simpler and avoids depending on the pool layer. Outcome: ✓ Good — clean separation between codegen and runtime.
- **Native `duckdb` driver for codegen, ADBC for queries.** Outcome: ✓ Good — codegen tooling stays separate from the runtime stack.

**Issues Resolved:**

- DuckDB `semantic_view()` table function emits valid SQL across grouped, row-level, and WHERE/ORDER BY/LIMIT-wrapped queries (924 unit tests).
- Arrow output exposed without forcing user-side ADBC cursor handling.
- All MockPool references removed; tests now run against real DuckDB.
- `duckdb-semantic-views 0.8.0` upstream fix unblocked the previously-xfail tests.
- `pip install semolina[duckdb]` and `semolina[all]` correctly install duckdb + pyarrow.

**Issues Deferred:**
