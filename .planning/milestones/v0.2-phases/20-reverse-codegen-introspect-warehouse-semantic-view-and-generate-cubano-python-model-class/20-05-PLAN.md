    The page describes: given a warehouse semantic view you've already created, use `cubano codegen` to

    Title: "How to generate Cubano model classes from warehouse views"

    Structure (how-to guide format per Diataxis — goal-oriented, illustrative, not step-by-step tutorial):

    ## Run codegen

    Show the basic command:
    ```bash
    cubano codegen my_schema.sales_view --backend snowflake
    ```
    Brief one-liner: introspects `my_schema.sales_view` and prints a Python class to stdout.

    ## Introspect multiple views at once

    ```bash
    cubano codegen schema.sales_view schema.orders_view --backend databricks
    ```
    All classes in one output block, single shared imports section.

    ## Pipe output to a file

    ```bash
    cubano codegen my_schema.sales_view --backend snowflake > models.py
    ```
    Brief note: no --output flag; redirect stdout.

    ## Choose a backend

    Two backends supported. Use `--backend` or `-b`:

    | Value | Warehouse | Introspects via |
    |-------|-----------|-----------------|
    | `snowflake` | Snowflake Semantic Views | `SHOW COLUMNS IN SEMANTIC VIEW` |
    | `databricks` | Databricks Metric Views | `DESCRIBE TABLE EXTENDED AS JSON` |

    Credentials are loaded from the same environment variables as the query engine (e.g. CUBANO_SNOWFLAKE_ACCOUNT etc).
    Link to backend connection guides for credential setup.

    ## Understand the generated output

    Use tabbed content for Snowflake vs Databricks:

    Given a Snowflake semantic view `analytics.sales_view` with a metric `revenue` (NUMBER),
    a dimension `country` (VARCHAR), and a fact `unit_price` (NUMBER with decimal):

    === "Snowflake"

        ```bash
        cubano codegen analytics.sales_view --backend snowflake
        ```

        Output:

        ```python
        from cubano import SemanticView, Metric, Dimension, Fact

        class SalesView(SemanticView, view="analytics.sales_view"):
            revenue = Metric()
            country = Dimension()
            unit_price = Fact()
        ```

    === "Databricks"

        Given a Databricks metric view `main.analytics.orders_view` with a measure `total_orders`
        (BIGINT) and a dimension `region` (STRING):

        ```bash
        cubano codegen main.analytics.orders_view --backend databricks
        ```

        Output:

        ```python
        from cubano import SemanticView, Metric, Dimension, Fact

        class OrdersView(SemanticView, view="main.analytics.orders_view"):
            total_orders = Metric()
            region = Dimension()
        ```

    Brief note after tabs: Databricks has no native Fact type — all non-measure fields map to `Dimension()`.

    ## Understand field type mapping

    Table showing the mapping:

    | Warehouse classification | Generated field type |
    |--------------------------|---------------------|
    | Metric / Measure | `Metric()` |
    | Dimension | `Dimension()` |
    | Fact (Snowflake only) | `Fact()` |

    ## Handle TODO comments

    When a field's SQL type has no clean Python equivalent (e.g. GEOGRAPHY, VARIANT, ARRAY, MAP, STRUCT),
    codegen emits a TODO comment:

    ```python
    # TODO: no clean Python type for GEOGRAPHY field "territory"
    territory = Dimension()
    ```

    Review and handle these manually after generation.

    ## Use a custom backend

    `--backend` also accepts a dotted Python import path for custom Engine subclasses:

    ```bash
    cubano codegen my_schema.my_view --backend my_package.backends.MyCustomEngine
    ```

    The class must implement `introspect(view_name: str) -> IntrospectedView`.
    It is instantiated with no arguments; credentials are the class's responsibility.

    ## Exit codes

    | Code | Meaning |
    |------|---------|
    | `0` | Success |
    | `1` | View not found, connection error, or invalid backend |

    ## See also

    - [Defining models](models.md) — model class structure and field types
    - [Snowflake connection](backends/snowflake.md) — credential setup for Snowflake
    - [Databricks connection](backends/databricks.md) — credential setup for Databricks
