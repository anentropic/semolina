# Phase 10.1: Refactor Query Interface to Model-Centric - Context

**Gathered:** 2026-02-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the Cubano query API from its current procedural form to be model-centric and more intuitive. No backward compatibility needed (library not yet released). Focus on getting the interface right for user ergonomics, IDE autocomplete, type safety, and discoverability.

</domain>

<decisions>
## Implementation Decisions

### Query Construction API

**Entry Point:**
- Users start queries via model class method: `users.query()`
- Model feels active/native; query builder chain flows naturally from the model

**Field Selection:**
- Users select fields using model attributes: `query.metrics(Users.revenue, Users.users_count)`
- Provides IDE autocomplete, catches typos at dev time, enforces type safety
- Attribute access discovered via IDE hints first (primary discovery method)

**Filtering:**
- Use operator overloads for WHERE conditions: `.where(Users.country == 'US')`
- Simple boolean logic over Q-object patterns; more Pythonic
- Support OR composition via operator: `(Users.country == 'US') | (Users.country == 'CA')`

### Result Handling & Execution

**Execution Model:**
- Eager execution: `.execute()` returns results immediately
- More predictable for warehouse queries, avoids connection lifecycle issues
- Explicit control over when warehouse is hit (no deferred errors during iteration)

**Result Format:**
- Custom Result object (typed, with helper methods)
- Preserves row access pattern already implemented in codebase
- Result object allows `.to_pandas()`, aggregation helpers, etc. as extensions

**Row Access:**
- Preserve existing row access implementation (already working)
- Both attribute and dict-like access if already supported

### Model Introspection

**Discovery Methods:**
- Primary: IDE autocomplete (Users.revenue shows up in hints while typing)
- Fallback: Explicit `.metrics()` and `.dimensions()` methods return Field objects

**Field Introspection Methods:**
- `Users.metrics()` — returns List[Field] with metadata (type, docstring, etc.)
- `Users.dimensions()` — returns List[Field] with metadata
- REPL-friendly: users can run `Users.metrics()` to see available options
- Preserve user's own docstring on model class (don't inject field lists)

**Field Objects:**
- Return rich Field objects (not just strings) from .metrics() / .dimensions()
- Include: field name, type, aggregation type (for metrics), docstring, warehouse-specific metadata

### Claude's Discretion

- Exact descriptor / __getattr__ implementation for graceful fallbacks
- Row serialization format and performance optimizations
- Helper methods on Result object (.summary(), .pivot(), etc.)
- IDE stub generation for autocomplete enhancement

</decisions>

<specifics>
## Specific Ideas

- Strong typing throughout the API is a stated project goal — validate toward it (even if this refactor doesn't perfect it)
- Library not yet released, so we optimize for "getting it right" over backward compatibility
- No backward compatibility migration path needed — full freedom to reshape the interface

</specifics>

<deferred>
## Open Questions for Research/Planning

**Namespace collision handling** — How to handle conflicts between built-in methods (query, metrics, dimensions) and user-defined field names?

Options to research:
1. Descriptors + __getattr__ — Pythonic but complex, type-checking challenges, IDE confusion
2. Reserved method names — Simpler, clearer (document: "don't name a field 'query'"), but less flexible
3. _fields dict approach — Explicit and safe, but less pretty

This decision should be researched and recommended by the planning phase; deferred from context gathering.
