  <action>
    Create `docs/guides/warehouse-testing.md` as a complete developer guide. The guide must cover all topics required for a developer to add warehouse tests in future phases.

    **Required sections (in order):**

    1. **Overview** (2-3 paragraphs)
       - Explain the recording/replay pattern: tests run against real warehouses once locally (with `--snapshot-update`), snapshots are committed, CI replays from committed snapshots using MockEngine — zero warehouse cost in CI.
       - Explain the fixture design: `snowflake_engine` and `databricks_engine` fixtures each independently handle their backend. The `backend_engine` parametrized fixture runs a single test function against both backends — pytest creates `[snowflake_engine]` and `[databricks_engine]` variants automatically.
       - Key benefit: tests are DRY (one function, two backends), no env vars needed to switch backends, snapshot entries are automatically tagged by backend in the `.ambr` file.

    2. **Quick Start: Adding a Snapshot Test** (step-by-step)
       - Step 1: Add your SemanticView model to `tests/test_snapshot_queries.py` (or a new test file)
       - Step 2: Update `SNAPSHOT_TEST_DATA` in `tests/conftest.py` to include the rows your view will expose
       - Step 3: Write the test function using `backend_engine` and `snapshot` fixtures:
         ```python
         def test_my_query(backend_engine, snapshot):
             """Validates my query returns expected results."""
             result = MyView.query().using("test").metrics(MyView.revenue).order_by(MyView.revenue).execute()
             rows = [dict(row.items()) for row in result]
             assert rows == snapshot
         ```
       - Step 4: Record the snapshot against both warehouses (credentials required):
         ```bash
         pytest --snapshot-update tests/test_snapshot_queries.py
         ```
         If only Snowflake credentials are available, Databricks variants are skipped (re-record with Databricks credentials when available).
       - Step 5: Verify the snapshot diff: `git diff tests/__snapshots__/`
       - Step 6: Commit: `git add tests/__snapshots__/ && git commit -m "feat: add snapshot test for ..."`

    3. **Recording Workflow** (detail)
       - How `--snapshot-update` works: `snowflake_engine` uses SnowflakeEngine if SNOWFLAKE_* credentials are set, otherwise skips. `databricks_engine` uses DatabricksEngine if DATABRICKS_* credentials are set, otherwise skips.
       - **Snowflake recording:** Required env vars: `SNOWFLAKE_ACCOUNT`, `SNOWFLAKE_USER`, `SNOWFLAKE_PASSWORD`, `SNOWFLAKE_WAREHOUSE`, `SNOWFLAKE_DATABASE`
       - **Databricks recording:** Required env vars: `DATABRICKS_HOST`, `DATABRICKS_HTTP_PATH`, `DATABRICKS_ACCESS_TOKEN`, `DATABRICKS_CATALOG`, `DATABRICKS_SCHEMA`
       - No `CUBANO_SNAPSHOT_BACKEND` env var needed — both backends are always recorded in one `pytest --snapshot-update` run (each skips gracefully if its credentials are missing)
       - What gets written to `.ambr` files: list[dict] representation of query results, tagged with `[snowflake_engine]` or `[databricks_engine]` suffix

    4. **Replay Mode (CI)**
       - No credentials needed for either backend
       - Both `snowflake_engine` and `databricks_engine` use MockEngine loaded with `SNAPSHOT_TEST_DATA` from `tests/conftest.py`
       - All 12 snapshot test variants pass in CI automatically as part of `pytest tests/`
       - `--snapshot-warn-unused` in CI warns about stale snapshots

    5. **Re-Recording After Changes**
       - When to re-record: SQL generation changes, SNAPSHOT_TEST_DATA changes, warehouse schema changes
       - How to re-record: `uv run pytest tests/test_snapshot_queries.py --snapshot-update` (requires credentials for the backends you want to update)
       - Review the git diff before committing: `git diff tests/__snapshots__/`
       - Any snapshot change in a PR should be reviewed intentionally

    6. **Stale Snapshot Cleanup**
       - When a test is removed, its snapshot entries (both `[snowflake_engine]` and `[databricks_engine]`) become stale
       - Detection: CI prints warning via `--snapshot-warn-unused`
       - Cleanup: run `pytest --snapshot-update` locally (syrupy removes unused entries automatically)
       - Manual alternative: delete the stale `# name: test_removed_test[...]` blocks from the `.ambr` file

    7. **Writing Good Snapshot Tests** (best practices)
       - Always use `.using("test")` to select the engine registered by the fixture
       - Always use `.order_by()` — warehouse does not guarantee row order without ORDER BY
       - Always convert to `list[dict]`: `rows = [dict(row.items()) for row in result]`
       - Use `.limit(N)` for large datasets to keep `.ambr` files small (≤ 20 rows recommended)
       - Use integer values in SNAPSHOT_TEST_DATA to avoid Decimal precision drift
       - Keep SNAPSHOT_TEST_DATA ≤ 10 rows per view
       - Use synthetic, controlled data — not production data
       - List `backend_engine` first in test signature to ensure engine is registered before queries execute

    8. **Snapshot File Format**
       - Brief explanation of Amber format (`.ambr`)
       - Show an example snippet of what the file looks like for one test with both backend variants

    9. **Troubleshooting**
       - "AssertionError: snapshot missing" → Run `pytest --snapshot-update` locally and commit
       - "Snapshot mismatch" after refactor → Re-record with `--snapshot-update`, review diff
       - "Test SKIPPED[snowflake_engine] in recording mode" → Check `SNOWFLAKE_*` environment variables are set
       - "Test SKIPPED[databricks_engine] in recording mode" → Check `DATABRICKS_*` environment variables are set
       - "Only one backend variant in .ambr" → The other backend's credentials were missing during recording; re-run `--snapshot-update` with the missing credentials

    **Style requirements:**
    - Use MkDocs Material admonitions where appropriate: `!!! tip`, `!!! warning`, `!!! note`
    - Code blocks with language tags: ` ```bash`, ` ```python`
    - Keep the guide practical and focused on the developer workflow
    - Do not include warehouse-specific DDL (the synthetic view setup is handled by fixtures)
    - Line length: match existing docs style (≤ 100 chars preferred in prose)

    **Do NOT include:**
    - Custom diff tooling instructions (deferred per locked decisions)
    - Performance regression testing (deferred)
    - `CUBANO_SNAPSHOT_BACKEND` env var (removed from design)
    - `CUBANO_SNAPSHOT_FORCE_MOCK` env var (removed from design)
