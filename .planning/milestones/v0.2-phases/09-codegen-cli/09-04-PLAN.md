### Task 3: Verify CODEGEN-05 is addressed appropriately

Per the research doc, CODEGEN-05 ("Generated code passes syntax validation before
writing") conflicts with the locked decision to trust template correctness.
Resolution: interpret CODEGEN-05 as validating the *input* Python files (not SQL output).

The `validate_python_syntax()` function in 09-03 already handles this via `ast.parse`.
Add a test to explicitly document this:

```python
class TestInputValidation:
    """CODEGEN-05: Input Python syntax is validated before processing."""

    def test_invalid_python_syntax_caught_before_output(self, tmp_path: Path) -> None:
        """No output written when Python syntax is invalid."""
        bad_file = tmp_path / "syntax_error.py"
        bad_file.write_text("def broken(:pass\n")
        output_file = tmp_path / "output.sql"

        result = runner.invoke(
            app,
            ["codegen", str(bad_file), "--backend", "snowflake", "--output", str(output_file)],
        )
        assert result.exit_code == 1
        # Output file must NOT be created when validation fails
        assert not output_file.exists()
```

This test is important: it ensures that when `--output` is specified, we do not
create a partial output file if validation fails. The generator must collect errors
first and only write output if there are no errors. Verify the generator in 09-03
works this way: errors are raised before any file write.

### Task 4: Verify CODEGEN-06 is addressed (circular relationships)

Per the context and research docs, CODEGEN-06 circular relationship validation is
delegated to the Cubano model API — `SemanticView.__init_subclass__` prevents

to re-validate.

Add a test documenting this design decision:

```python
class TestCircularRelationshipDesign:
    """CODEGEN-06: Circular relationship validation handled by model API."""

    def test_model_api_prevents_circular_deps_at_definition(self) -> None:

        Codegen trusts the API and does not re-validate relationships.
```

```python
"""Tests for CLI utility functions."""

from pathlib import Path
from cubano.cli.utils import resolve_input_paths
import sys
import io
from rich.console import Console


def make_test_stderr() -> Console:
    return Console(file=io.StringIO(), stderr=True)


class TestResolveInputPaths:
    def test_explicit_py_file(self, tmp_path: Path) -> None:
        f = tmp_path / "models.py"
        f.write_text("")
        result = resolve_input_paths(str(f), make_test_stderr())
        assert result == [f.resolve()]

    def test_nonexistent_file_raises_exit(self, tmp_path: Path) -> None:
        import typer
        with pytest.raises(typer.Exit):
            resolve_input_paths(str(tmp_path / "nonexistent.py"), make_test_stderr())

    def test_directory_scans_py_files(self, tmp_path: Path) -> None:
        (tmp_path / "a.py").write_text("")
        (tmp_path / "b.py").write_text("")
        (tmp_path / "other.txt").write_text("")
        result = resolve_input_paths(str(tmp_path), make_test_stderr())
        names = {p.name for p in result}
        assert "a.py" in names
        assert "b.py" in names
        assert "other.txt" not in names

    def test_directory_excludes_pycache(self, tmp_path: Path) -> None:
        pycache = tmp_path / "__pycache__"
        pycache.mkdir()
        (pycache / "cached.py").write_text("")
        (tmp_path / "real.py").write_text("")
        result = resolve_input_paths(str(tmp_path), make_test_stderr())
        assert all("__pycache__" not in str(p) for p in result)
```

### Task 6: Run the full test suite and fix any integration issues

```bash
uv run --extra dev pytest tests/ -v
```

Common integration issues to watch for:

1. **`CliRunner(mix_stderr=False)` output separation**: Typer routes some error
   messages to stderr, not stdout. With `mix_stderr=False`, `result.output` is
   only stdout. Use `result.stdout` for SQL and check exit codes for errors.
   If tests fail because expected text is in stderr, use `result.stderr` instead.

2. **Rich Console in CliRunner context**: CliRunner captures stdout/stderr as
   strings. Rich's Console may detect non-TTY and disable colors. This is correct
   behavior — assert on plain text, not ANSI codes.

3. **`typer.Exit` propagation through CliRunner**: CliRunner catches `SystemExit`;
   check `result.exit_code`, not whether an exception was raised.

4. **Generator raises `typer.Exit(1)` on errors**: This propagates through CliRunner.
   Verify the exit code is exactly 1 (not 2, which is a Typer usage error).

5. **Template files not found after `uv sync`**: If `Path(__file__).parent / "templates"`
   fails in tests, it may be a path resolution issue. Confirm templates are colocated
   with `renderer.py` in `src/cubano/codegen/templates/`.
