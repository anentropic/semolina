# Phase 13.1: Implement filter lookup system and WHERE clause compiler - Research

**Researched:** 2026-02-22
**Domain:** SQL filter compilation, typed predicate tree IR, Python structural pattern matching
**Confidence:** HIGH

## Summary

Phase 13.1 replaces the string-keyed Q-object filter IR with a typed predicate tree (`Predicate` / `Lookup[T]` hierarchy), adds named filter methods to Field (`.between()`, `.in_()`, `.like()`, etc.), and compiles predicate trees into parameterized SQL WHERE clauses. The architecture is fully specified in `_ideas/filter-api-design.md`.

The current codebase has a clear, well-bounded surface area for this change. The Q class in `filters.py` (~130 lines) is replaced entirely. Field operators in `fields.py` (6 methods, ~120 lines) are rewritten to return typed Predicate nodes instead of Q objects. The `_build_where_clause()` stub in `engines/sql.py` (currently returns `"WHERE 1=1"`) becomes the real compiler. The query builder's `_filters` field type changes from `Q | None` to `Predicate | None`. All existing tests for Q objects and field operators must be rewritten, and the public API export of `Q` from `cubano.__init__` is replaced with `Predicate`.

This is a pure internal refactor with one public API change: `Q` is removed and `Predicate` is exported. The `.where()` method signature changes from accepting `Q | None` to `Predicate | None`. All other public interfaces (`.execute()`, `.to_sql()`, Model.query()) remain unchanged.

**Primary recommendation:** Implement in three layers bottom-up: (1) predicate tree types in `filters.py`, (2) Field methods returning predicates in `fields.py`, (3) WHERE compiler in `engines/sql.py` with parameterized output. Update `query.py`, `__init__.py`, all tests, docs, and jaffle-shop in coordinated steps.

### Pattern 1: Predicate Tree (Composite Pattern)
**What:** Algebraic data type using dataclasses -- `Predicate` base with `And`, `Or`, `Not` composites and `Lookup[T]` leaves.
**When to use:** Every filter expression builds a tree of these nodes.
**Example:**
```python
# filters.py -- the complete IR (from design doc)
from __future__ import annotations
from dataclasses import dataclass
from typing import Any, Generic, TypeVar

T = TypeVar("T")

class Predicate:
    """Base for all filter predicates. Supports &, |, ~ composition."""
    def __and__(self, other: Predicate) -> And:
        return And(left=self, right=other)
    def __or__(self, other: Predicate) -> Or:
        return Or(left=self, right=other)
    def __invert__(self) -> Not:
        return Not(inner=self)

@dataclass(frozen=True)
class And(Predicate):
    left: Predicate
    right: Predicate

@dataclass(frozen=True)
class Or(Predicate):
    left: Predicate
    right: Predicate

@dataclass(frozen=True)
class Not(Predicate):
    inner: Predicate

@dataclass(frozen=True)
class Lookup(Predicate, Generic[T]):
    """Typed leaf node. Subclass for each operation."""
    field_name: str
    value: T

class Exact(Lookup[Any]): ...
class Gt(Lookup[Any]): ...
# ... etc
```
Source: `_ideas/filter-api-design.md`

### Pattern 2: Pattern-Matching Compiler
**What:** `match/case` on predicate node types to emit SQL fragments + params.
**When to use:** `_build_where_clause()` in SQLBuilder.
**Example:**
```python
# engines/sql.py
def _compile_predicate(self, node: Predicate) -> tuple[str, list[Any]]:
    match node:
        case Exact(field_name=f, value=v):
            return f"{self.dialect.quote_identifier(f)} = %s", [v]
        case Gt(field_name=f, value=v):
            return f"{self.dialect.quote_identifier(f)} > %s", [v]
        case In(field_name=f, value=values) if not values:
            return "1 = 0", []  # empty IN -> always false
        case In(field_name=f, value=values):
            placeholders = ", ".join(["%s"] * len(values))
            return f"{self.dialect.quote_identifier(f)} IN ({placeholders})", list(values)
        case Between(field_name=f, value=(lo, hi)):
            return f"{self.dialect.quote_identifier(f)} BETWEEN %s AND %s", [lo, hi]
        case IsNull(field_name=f, value=True):
            return f"{self.dialect.quote_identifier(f)} IS NULL", []
        case IsNull(field_name=f, value=False):
            return f"{self.dialect.quote_identifier(f)} IS NOT NULL", []
        case And(left=l, right=r):
            l_sql, l_params = self._compile_predicate(l)
            r_sql, r_params = self._compile_predicate(r)
            return f"({l_sql} AND {r_sql})", l_params + r_params
        case Or(left=l, right=r):
            l_sql, l_params = self._compile_predicate(l)
            r_sql, r_params = self._compile_predicate(r)
            return f"({l_sql} OR {r_sql})", l_params + r_params
        case Not(inner=i):
            i_sql, i_params = self._compile_predicate(i)
            return f"NOT ({i_sql})", i_params
        case Lookup():
            raise NotImplementedError(
                f"Unsupported lookup type: {type(node).__name__}"
            )
        case _:
            raise TypeError(f"Expected Predicate, got {type(node).__name__}")
```
Source: `_ideas/filter-api-design.md` compiler sketch

### Pattern 3: Parameterized Execution Pipeline
**What:** Compiler returns `(sql_template, params)`, engines pass both to `cursor.execute(sql, params)`.
**When to use:** Every query with filters in `execute()`.
**Example:**
```python
# engines/snowflake.py -- execute() with params
sql, params = self._build_sql_with_params(query)  # returns template + params
cur.execute(sql, params)  # Snowflake connector binds params safely

# Query.to_sql() -- renders inline for display
sql, params = builder.build_select_with_params(query)
return builder.render_inline(sql, params)  # substitute %s with repr(value)
```

### Anti-Patterns to Avoid
- **String interpolation in SQL:** Never f-string or `.format()` user values into SQL. The compiler MUST use parameterized placeholders.
- **Catch-all Lookup without NotImplementedError:** The `case Lookup()` catch-all must raise `NotImplementedError` for unknown lookup subclasses, not silently ignore them.
- **Mutable predicate nodes:** All dataclasses must be `frozen=True` to match the immutable query builder pattern.
- **Re-implementing Q composition logic:** The `Predicate.__and__`, `__or__`, `__invert__` operators handle composition. Do NOT duplicate logic in `_Query.where()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SQL injection prevention | Manual string escaping | DB driver bind params via `cursor.execute(sql, params)` | Drivers handle escaping for every data type, every edge case |
| Pattern matching dispatch | if/elif chain on isinstance | `match/case` structural pattern matching | Exhaustiveness, readability, direct field extraction |
| Predicate composition | Custom tree-building methods | `__and__`, `__or__`, `__invert__` on Predicate base | Python operators give natural `(a & b) \| ~c` syntax |

**Key insight:** The compiler is ~100-150 lines of match/case dispatch. The complexity is in getting the parameterization right across all backends and edge cases, not in the tree structure.

## Common Pitfalls

### Pitfall 1: Forgetting to propagate params through composite nodes
**What goes wrong:** `And(left, right)` compiles SQL correctly but loses params from one branch.
**Why it happens:** Easy to concatenate SQL strings but forget to concatenate param lists.
**How to avoid:** Every `_compile_predicate` call returns `(sql, params)` tuple. Composite nodes concatenate BOTH.
**Warning signs:** Tests pass for simple predicates but fail for `(a & b)` compositions.

### Pitfall 2: Placeholder style mismatch across backends
**What goes wrong:** Compiler emits `%s` but DuckDB expects `?`. Or vice versa.
**Why it happens:** Each DB driver uses a different PEP 249 `paramstyle`.
**How to avoid:** Compiler always emits a canonical placeholder (e.g. `%s`). Each engine's `execute()` translates to its driver's style before calling `cursor.execute()`. Alternatively, the `Dialect` class exposes a `placeholder` property and the compiler uses it directly.
**Warning signs:** `ProgrammingError` at execution time mentioning unexpected `%s` or `?`.
**Verified placeholders:**
- Snowflake connector: `%s` (format style, default), also supports `?` (qmark) and `%(name)s` (pyformat). Source: Context7/Snowflake docs.
- Databricks connector: `?` (qmark), `:name` (named), `%(name)s` (pyformat, legacy). Source: Context7/databricks-sql-python docs.
- DuckDB: `?` (qmark), `$1` (positional), `$name` (named). Source: Context7/DuckDB docs.

### Pitfall 3: `from __future__ import annotations` breaking match/case
**What goes wrong:** With `from __future__ import annotations`, all annotations become strings. If match/case guard clauses or structural patterns use annotated types, they may behave unexpectedly.
**Why it happens:** `from __future__ import annotations` defers annotation evaluation. Match/case patterns work on runtime values, not annotations, so this is actually safe for pattern matching. The real risk is in `isinstance` checks within `Predicate.__and__` etc.
**How to avoid:** The match/case in the compiler matches on dataclass instances (runtime objects), not annotations. This is safe. BUT: the `Predicate.__and__` etc. methods use `other: Predicate` annotation -- with `from __future__ import annotations`, this is fine because it's just a type hint, not a runtime check. No action needed, but worth verifying.
**Warning signs:** None expected -- this is a "verify once" pitfall.

### Pitfall 4: basedpyright strict mode and Generic[T] inheritance
**What goes wrong:** `class Lookup(Predicate, Generic[T])` may trigger basedpyright errors about multiple base classes or Generic usage.
**Why it happens:** basedpyright strict mode is very particular about Generic inheritance patterns.
**How to avoid:** Test the exact class hierarchy with `uv run basedpyright` early. The design doc uses `class Lookup(Predicate, Generic[T])` which is standard Python -- should work. If issues arise, `Lookup` can use `T = TypeVar("T")` with explicit `value: T` on each subclass.
**Warning signs:** basedpyright errors mentioning `reportIncompatibleVariableOverride` or `reportGeneralClassIssues`.

### Pitfall 5: `__eq__` and `__hash__` on Predicate base class
**What goes wrong:** Defining `__eq__` on `Predicate` (inherited from dataclass) but not `__hash__` breaks set/dict usage. Or: existing `Field.__eq__` returns `Predicate` (not `bool`) which conflicts with `Predicate.__eq__` from dataclass.
**Why it happens:** Python sets `__hash__ = None` when `__eq__` is overridden without explicit `__hash__`.
**How to avoid:** `Predicate` is NOT a dataclass itself (it's a plain class with operator methods). Only `And`, `Or`, `Not`, `Lookup` are frozen dataclasses. `Field.__eq__` already returns `Q` (will change to `Exact`), and `Field.__hash__ = object.__hash__` is already set. This pattern carries forward unchanged.
**Warning signs:** `TypeError: unhashable type` errors.

### Pitfall 6: WHERE clause omitted when _filters is None
**What goes wrong:** Forgetting to handle the `_filters is None` case in the parameterized pipeline, leading to `WHERE` being added with no conditions.
**Why it happens:** The current code already handles this (`if query._filters is not None`), but the new parameterized pipeline needs the same guard.
**How to avoid:** The `build_select` method already conditionally appends WHERE. Ensure the parameterized version follows the same pattern -- only add WHERE clause when `_filters` is not None.
**Warning signs:** SQL output contains bare `WHERE` with no conditions.

### Pitfall 7: LIKE/ILIKE pattern injection
**What goes wrong:** User passes `%` or `_` characters as filter values for `.like()` and they're interpreted as SQL wildcards.
**Why it happens:** LIKE patterns use `%` and `_` as wildcards. If the user value contains these characters and is used as a bind parameter, the DB driver handles the value correctly (it's just a string). But the `.startswith("x")` and `.endswith("x")` lookups need to prepend/append `%` wildcards -- these wildcards should be part of the parameter value, not the SQL template.
**How to avoid:** For `.startswith("x")`, compile to `field LIKE %s` with param `"x%"` (wildcard in the param). For `.like("pat")`, compile to `field LIKE %s` with param `"pat"` (user provides their own wildcards). Document this clearly.
**Warning signs:** Tests for `.startswith("ab%cd")` matching unexpected rows.

```python
# fields.py
def __eq__(self, value: Any) -> Exact:
    return Exact(field_name=self.name, value=value)

def __ne__(self, value: Any) -> Not:
    return Not(Exact(field_name=self.name, value=value))

def between(self, lo: Any, hi: Any) -> Between:
    return Between(field_name=self.name, value=(lo, hi))

def in_(self, values: list[Any]) -> In:
    return In(field_name=self.name, value=values)

def isnull(self) -> IsNull:
    return IsNull(field_name=self.name, value=True)

def lookup(self, lookup_cls: type[Lookup[T]], value: T) -> Lookup[T]:
    return lookup_cls(field_name=self.name, value=value)
```

    """Render SQL with params substituted inline. For display/debugging only."""

### Parameterization Strategy

The design decision specifies:
1. Compiler emits `%s` placeholders internally
2. Each engine adapter translates to its driver's format
3. `to_sql()` renders params inline for display

**Recommended implementation for placeholder translation:**

Add a `placeholder` property to `Dialect`:
- `SnowflakeDialect.placeholder = "%s"` (Snowflake default paramstyle is `format`)
- `DatabricksDialect.placeholder = "?"` (Databricks default paramstyle is `qmark`)
- `MockDialect.placeholder = "%s"` (for consistency with Snowflake)

The compiler uses `self.dialect.placeholder` instead of hardcoded `%s`. This avoids a post-compilation string replacement step and is cleaner. The engine's `execute()` then passes params directly to `cursor.execute()`.

**Alternative:** Compiler always emits `%s`, engine replaces `%s` with `?` before passing to cursor. Simpler compiler, slightly more fragile (what if `%s` appears in a string literal?). Since string literals are always bind parameters (never interpolated), this risk is negligible. The dialect property approach is recommended because it's more explicit and handles future backends cleanly.

### where() Varargs Transition

Current signature: `def where(self, condition: Q | None) -> _Query`
New signature: `def where(self, *conditions: Predicate | None) -> _Query`

The varargs signature means:
- `.where(a, b, c)` is sugar for `.where(a).where(b).where(c)` (all ANDed)
- `.where(None)` is still a no-op
- `.where(a, None, b)` ignores None values, ANDs `a` and `b`
- `.where()` with no args is a no-op (returns self)

## Open Questions

1. **Placeholder escaping for LIKE wildcards in `.startswith()` / `.endswith()`**
   - What we know: `.startswith("x")` compiles to `LIKE 'x%'` -- the `%` wildcard must be in the param value, not the SQL template. If the user value itself contains `%` or `_`, these are literal characters in the context of bind parameters (the DB driver escapes them).
   - What's unclear: Should `.startswith("ab%cd")` match rows starting with literal `ab%cd`? In standard SQL, `LIKE 'ab%cd%'` would match `ab` + anything + `cd` + anything. But with bind params, the value is `"ab%cd%"` which the driver treats as a literal pattern to the LIKE operator -- meaning `%` IS still a wildcard in LIKE context even as a bind parameter. The driver doesn't escape LIKE wildcards.
   - Recommendation: For `.startswith()` / `.endswith()`, escape `%` and `_` in the user value before appending the wildcard. Use a helper: `value.replace("%", "\\%").replace("_", "\\_")` + `"%"`. Add `ESCAPE '\\'` to the SQL. This matches Django's approach. For `.like()` / `.ilike()`, the user provides the full pattern including wildcards -- no escaping.

2. **`to_sql()` inline rendering of complex types**
   - What we know: `to_sql()` substitutes params into SQL template for display. Simple values (`str`, `int`, `float`) are straightforward with `repr()`.
   - What's unclear: How to render `date`, `datetime`, `Decimal` values inline. `repr(date(2025, 1, 1))` gives `datetime.date(2025, 1, 1)` which is not valid SQL.
   - Recommendation: For v0.2, use `repr()` for all types in `to_sql()`. This is for debugging only, not execution. Document that `to_sql()` output is not guaranteed to be executable SQL. Revisit in v0.3 if needed.

3. **Predicate `__repr__` for debugging**
   - What we know: Q objects had a readable `__repr__`. Predicate tree nodes need similar.
   - What's unclear: Exact format for nested trees.
   - Recommendation: Use dataclass auto-generated `__repr__` (from `@dataclass(frozen=True)`). This gives `And(left=Exact(field_name='country', value='US'), right=Gt(field_name='revenue', value=1000))` which is verbose but accurate. Add a custom `__repr__` to `Predicate` if the auto-generated one is too verbose -- but start with auto-generated.
