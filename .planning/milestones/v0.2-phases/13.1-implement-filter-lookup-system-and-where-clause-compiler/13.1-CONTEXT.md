# Phase 13.1: Implement filter lookup system and WHERE clause compiler - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the string-keyed Q-object filter IR with a typed predicate tree (`Predicate` / `Lookup[T]` hierarchy), add named filter methods to Field (`.between()`, `.in_()`, `.like()`, etc.), and compile predicate trees into parameterized SQL WHERE clauses. The architecture is fully specified in `_ideas/filter-api-design.md`.

</domain>

<decisions>
## Implementation Decisions

### `.where()` chaining API
- Keep AND chaining: `.where(a).where(b)` means `And(a, b)`, same behavior as today
- Accept varargs: `.where(a, b, c)` is sugar for ANDing all arguments
- Keep `None` as no-op: `.where(None)` returns query unchanged (supports `where(pred if cond else None)` pattern)
- Type signature: `def where(self, *conditions: Predicate | None) -> _Query`

### Backwards compat / migration
- Delete Q class outright — no deprecation wrapper, clean break in this phase
- Update all imports, tests, and docs that reference Q
- Export `Predicate` from `cubano.__init__` (top-level import for type annotations)
- Lookup classes (Exact, Gt, Like, etc.) — Claude's discretion on whether to make public from `cubano.filters` or keep internal

### Compiler parameterization
- **HARD REQUIREMENT: Bind params everywhere. SQL injection must be impossible.**
- Compiler produces `(sql_template, params)` internally — e.g. `("WHERE country = %s", ("US",))`
- `execute()` passes template + params to DB connector (bind parameters, safe)
- `to_sql()` substitutes params into template and returns rendered `str` (display/debugging only)
- `to_sql()` return type stays `str` — no breaking change
- Placeholder style follows Django pattern: compiler emits `%s`, each engine adapter translates to its driver's format (`%s` for Snowflake, `?` for DuckDB, etc.)
- Identifiers (column/table names) quoted with dialect-specific quoting, not parameterized — safe because field names come from model definitions, never user input

### Edge case behavior
- Empty `.in_([])` compiles to `1 = 0` (always false) — follows Django precedent, no error raised
- Nested NOT (`~~predicate`) — explicit test coverage required
- Both edge cases must have dedicated tests

### Claude's Discretion
- Test granularity (per-lookup vs by-category grouping)
- Assertion style for compiler tests (SQL string matching vs structural)
- Whether lookup classes are public API from `cubano.filters` or internal
- Exact predicate tree simplification for double negation

</decisions>

<specifics>
## Specific Ideas

- "Follow Django's pattern" — `%s` placeholder internally, engine adapters translate for their driver
- "Follow Django for empty IN" — `IN ()` → `1 = 0` instead of raising
- Architecture fully specified in `_ideas/filter-api-design.md` — typed predicate tree, unified `Lookup[T]` base, pattern-matching compiler
- Bind params are non-negotiable for execution — `to_sql()` is the only place values render inline

</specifics>
