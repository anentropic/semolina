## User Constraints (from CONTEXT.md)

### Locked Decisions

#### Column name mapping strategy

- Python field name = `original_column_name.lower()` (e.g., `ORDER_ID` → `order_id`)
- Add `source: str | None = None` parameter to `Field.__init__()` (and therefore to
  `Metric`, `Dimension`, `Fact` which inherit it).
- `source=None` (default): the SQL builder defers to `dialect.normalize_identifier(field.name)`
  to get the SQL-facing column name. Each dialect knows its identifier folding:
  - `SnowflakeDialect.normalize_identifier(name)` → `name.upper()`
  - `DatabricksDialect.normalize_identifier(name)` → `name.lower()`
  - `MockDialect.normalize_identifier(name)` → identity (name unchanged)
- `source="some_value"` (explicit): the SQL builder uses that string verbatim (after quoting).
  This is how users specify explicitly-quoted, non-default-casing columns:
  - Snowflake quoted-lowercase column `"order_id"` → `source="order_id"` (verbatim, not uppercased)

#### When codegen emits source=

Codegen should emit `source=` only when it is necessary — i.e., when the dialect would
auto-normalize to a different name than the original:

- Rule: if `dialect.normalize_identifier(python_name) == original_column_name` → no `source=`
- Otherwise → emit `source=original_column_name`

Examples for Snowflake (normalize = uppercase):
- `ORDER_ID` → python `order_id`, normalize(`order_id`) = `ORDER_ID` = original → `order_id = Metric[int]()`
- `"order_id"` → python `order_id`, normalize(`order_id`) = `ORDER_ID` ≠ `order_id` → `order_id = Metric[int](source="order_id")`

This logic needs comprehensive test coverage and clear docs.

#### SQL builder changes

- Each `Dialect` gains a `normalize_identifier(name: str) -> str` method alongside the
  existing `quote_identifier`.
- SQL builder resolves the SQL column name as:
  ```python
  col_name = field.source if field.source is not None else dialect.normalize_identifier(field.name)
  sql_ref = dialect.quote_identifier(col_name)
  ```
- No backward compatibility shim needed — the new behavior is the correct behavior. Existing
  hand-crafted models without `source=` will now get `normalize_identifier` applied (which is
  what Snowflake users already expected to happen implicitly).
- View names in `FROM` / `GROUP BY ALL`: left untouched. User supplies the view name in the
  correct case. We document that view names are always quoted verbatim, so Snowflake users
  with unquoted view names must use all-caps in their model's `view=` parameter.

#### Generic[T] on Field

- `Field` becomes `Generic[T]` (using `typing.Generic`).
- `Metric`, `Dimension`, `Fact` inherit and expose `[T]` subscript via the generic parent.
- `__get__` overloads: return `Self` when `obj is None` (class access for query building),
  return `T` when `obj` is not None (instance access — currently unused in Cubano, but correct
  for type checkers).
- Generated code form: `order_id = Metric[int]()` — no explicit variable annotation; type
  checker infers `Metric[int]` from the descriptor.
- For unmapped/TODO SQL types: `order_id = Metric[Any]()  # TODO: GEOGRAPHY type`
- `import datetime` logic retained (drives `needs_datetime` for the template's `import datetime`).
- `from typing import Any` added to generated output when Any is used.

#### Exit codes on the codegen CLI

- Motivation: good CLI convention (not driven by current scripting needs).
- Distinct codes:
  - `1` — Unexpected/unhandled error (catch-all)
  - `2` — Invalid backend (unknown or unsupported `--backend` value)
  - `3` — View not found in the warehouse
  - `4` — Connection / authentication failure
- Documentation: exit codes table added to **both** the `--help` text and the `codegen` how-to
  guide page.
- Note: Typer currently emits exit code 2 for its own validation errors (missing args, etc.).
  The plan should check whether to align or work around Typer's built-in exit code 2.

## Summary

Phase 20.1 is a focused bug-fix and polish phase with three independent changes to the existing codebase. The changes are non-trivial but fully within the existing architecture; no new libraries or major structural additions are required.

**Gap 1: Column name case mismatch.** Snowflake's `SHOW COLUMNS IN VIEW` returns uppercase column names (e.g., `ORDER_ID`). The current engine code passes these uppercase strings directly to `IntrospectedField.name`, so generated Python field names are un-Pythonic (`ORDER_ID = Metric[int]()`). The fix is two-pronged: (a) the engine lowercases `column_name` before storing it in `IntrospectedField.name`, and (b) `Field` gains a `source: str | None` parameter that the SQL builder uses instead of `field.name` when generating column references. Each `Dialect` gains a `normalize_identifier()` method (Snowflake → uppercase, Databricks → lowercase, Mock → identity) used when `source` is absent. Codegen emits `source=` only when lowercasing would break SQL round-trip.

**Gap 2: Generic[T] on Field.** Making `Field` a `Generic[T]` is a pure Python typing enhancement. Runtime behavior is unchanged — `isinstance(f, Metric)` continues to work because Python's Generic machinery doesn't alter the MRO. The `Metric[int]()` syntax works at runtime and produces a real `Metric` instance. The overload pattern for `__get__` that satisfies basedpyright strict mode requires returning `Field[T]` (or subtype) on class access and `T` on instance access. The Jinja2 template needs updating to emit `Metric[int]()` instead of `Metric()`.

**Gap 3: Exit codes.** Typer already uses exit code 2 for its own argument validation errors (missing required options). Since the CONTEXT.md assigns code 2 to "invalid backend", these collide at the `--backend` level — but Typer's code 2 fires on missing `--backend`, while our code 2 fires after `--backend` is provided but the value is invalid. This is a non-collision in practice. The plan should implement distinct exit codes by replacing the current `raise typer.Exit(code=1)` calls with typed exception-based dispatch.

**Primary recommendation:** Implement in three independent plan waves: (1) `source=` on Field + `normalize_identifier` on Dialect + SQL builder update + codegen emit logic, (2) `Generic[T]` on Field + template update, (3) exit code refactor + docs.

### Pattern 1: Field as Generic[T] Descriptor

**What:** `Field` inherits from `Generic[T]`. Subclasses (`Metric`, `Dimension`, `Fact`) gain subscript syntax without any additional code. Python's `Generic.__class_getitem__` handles `Metric[int]` at both class definition time and instantiation time.

**Key verified fact:** `Metric[int]()` at runtime produces a real `Metric` instance. `isinstance(Sales.revenue, Metric)` continues to return `True` — Generic does not change MRO or isinstance semantics.

**`__get__` overload pattern (basedpyright strict compatible):**

```python
# Source: verified via uv run python, Python 3.14
from __future__ import annotations
from typing import TypeVar, Generic, overload, Any

T = TypeVar("T")

class Field(Generic[T]):
    def __init__(self, source: str | None = None) -> None:
        self.name: str | None = None
        self.source: str | None = source

    @overload
    def __get__(self, obj: None, cls: type) -> Field[T]: ...
    @overload
    def __get__(self, obj: object, cls: type) -> T: ...
    def __get__(self, obj: object | None, cls: type) -> Field[T] | T:
        if obj is None:
            return self
        raise AttributeError(
            f"Field '{self.name}' cannot be accessed from instances."
        )
```

**Basedpyright strict note:** The implementation signature `-> Field[T] | T` is the only form that satisfies both overloads without `type: ignore`. Using `-> Any` also works but is weaker. Test with `uv run basedpyright` after each change.

**`__hash__` must be preserved:** `fields.py` already has `__hash__ = object.__hash__` to preserve set/dict usage despite the `__eq__` override. This must remain after the Generic refactor.

### Pattern 2: normalize_identifier on Dialect

**What:** New abstract method on `Dialect` that maps a Python field name to the SQL-facing column name the warehouse stores without quoting.

```python
# In src/cubano/engines/sql.py
@abstractmethod
def normalize_identifier(self, name: str) -> str:
    """
    Return the SQL-facing column name for a given Python field name.

    Called when field.source is None. Each dialect knows its
    default identifier folding:
    - Snowflake: unquoted identifiers are stored UPPERCASE
    - Databricks: unquoted identifiers are stored lowercase
    - Mock: identity (no transformation)
    """
    pass
```

Concrete implementations:
- `SnowflakeDialect.normalize_identifier(name)` → `name.upper()`
- `DatabricksDialect.normalize_identifier(name)` → `name.lower()`
- `MockDialect.normalize_identifier(name)` → `name` (identity)

**SQL builder column resolution** (replaces bare `field.name` usage in `_build_select_clause` and `_build_order_by_clause`):

```python
def _resolve_col_name(self, field: Any) -> str:
    """Resolve SQL column name: source takes priority, else normalize."""
    assert field.name is not None
    if field.source is not None:
        return field.source
    return self.dialect.normalize_identifier(field.name)
```

Then in `_build_select_clause`:
```python
# Metrics
col_name = self._resolve_col_name(metric)
wrapped = self.dialect.wrap_metric(col_name)

# Dimensions/facts
col_name = self._resolve_col_name(dim)
quoted = self.dialect.quote_identifier(col_name)
```

**IMPORTANT — WHERE clause:** The WHERE clause compiler uses `field_name` from `Lookup` nodes (strings stored in predicates), NOT from `Field` descriptors directly. The predicate's `field_name` is set at the time the predicate is created (`__eq__`, `in_()`, etc.) from `field.name` (the Python attribute name). This means WHERE clause compilation also needs `normalize_identifier` applied. The current code in `_compile_predicate` calls `q(f)` where `f` is the field_name from the predicate — this is the Python field name without normalization. This must be addressed: predicate compilation should also normalize. However, the predicate only stores the Python field name, not the source. The SQL builder must normalize the field_name in the predicate via `dialect.normalize_identifier(f)` before quoting.

```python
# In _compile_predicate, replace:
case Exact(field_name=f, value=v):
    return f"{q(f)} = {ph}", [v]

# With:
case Exact(field_name=f, value=v):
    normalized = self.dialect.normalize_identifier(f)
    return f"{q(normalized)} = {ph}", [v]
```

This applies to all Lookup branches. There is no `source` in predicates — predicates always use the Python field name, which the SQL builder normalizes at compile time.

**Backward compatibility note:** Existing hand-crafted models (no `source=`) that worked on Snowflake before this change worked because Snowflake is case-insensitive for unquoted identifiers — `"order_id"` and `"ORDER_ID"` both match `ORDER_ID`. After this change, the generated SQL will use `"ORDER_ID"` (normalized), which is equally correct. No regressions expected.

### Pattern 5: Exit Codes

**Typer exit code collision analysis (verified):**
- Typer uses exit code `2` for its own validation errors (missing required args/options). This fires when `--backend` is entirely absent.
- Our proposed code `2` is for "invalid backend value" — fires when `--backend mysql` is passed (value provided but unrecognized).
- These do NOT collide at the same invocation — Typer's `2` fires before our logic runs.
- Recommendation: keep the proposed mapping as designed. Document in `--help` that code 2 means "invalid backend specifier" (distinct from Typer's missing-arg 2, which is a user-facing diff).

**Current CLI error handling:**
```python
# All errors currently use code=1
raise typer.Exit(code=1) from e
```

**New pattern — exception-based dispatch:**
```python
EXIT_INVALID_BACKEND = 2
EXIT_VIEW_NOT_FOUND = 3
EXIT_CONNECTION_ERROR = 4

try:
    engine = _resolve_backend(backend)
except typer.BadParameter as e:
    _stderr.print(f"[bold red]Error:[/bold red] {e}")
    raise typer.Exit(code=EXIT_INVALID_BACKEND) from e

for view_name in views:
    try:
        introspected = engine.introspect(view_name)
        ...
    except ViewNotFoundError as e:
        _stderr.print(f"[bold red]Error:[/bold red] {e}")
        raise typer.Exit(code=EXIT_VIEW_NOT_FOUND) from e
    except ConnectionError as e:
        _stderr.print(f"[bold red]Error:[/bold red] {e}")
        raise typer.Exit(code=EXIT_CONNECTION_ERROR) from e
    except RuntimeError as e:
        _stderr.print(f"[bold red]Error:[/bold red] {e}")
        raise typer.Exit(code=1) from e
```

**Issue:** Currently `introspect()` raises only generic `RuntimeError` for all failure modes. To distinguish "view not found" from "connection failure", the engine(s) must either raise distinct exception subtypes OR the CLI must inspect the error message. The cleaner solution is to introduce `CubanoViewNotFoundError` and `CubanoConnectionError` as exception subclasses, and have `SnowflakeEngine.introspect()` raise them. However, this requires touching the engine code.

**Simpler alternative (lower risk):** Keep `RuntimeError` for all engine errors but categorize them in the CLI via a keyword pattern match on the exception message. This is fragile for a library but acceptable for v0.1 given the context says "not driven by current scripting needs." The plan should decide which approach to take.

**Recommended decision:** Introduce two narrow exception classes in `cubano.engines.base` (`CubanoViewNotFoundError(RuntimeError)` and `CubanoConnectionError(RuntimeError)`) and have the Snowflake engine raise them. Databricks engine gets stubs. This is clean and aligns with good library design.

### Anti-Patterns to Avoid

- **Modifying `field.name` after `__set_name__`:** `field.name` must remain the Python attribute name. Never use it as the SQL column name directly — always go through `_resolve_col_name()`.
- **Adding `source` to `Lookup` predicates:** Predicates store `field_name` (Python name). SQL normalization is the SQL builder's job, not the predicate's job.
- **Using `Any` return type on `__get__` implementation:** While it silences type errors, it defeats the purpose of Generic. Use the union form `Field[T] | T`.
- **Template emitting `data_type` as `None`:** The template must handle `None` data_type (for TODO fields) — emit `Metric[Any]()` with a `# TODO: ...` comment when `data_type` is None.

## Common Pitfalls

### Pitfall 1: WHERE clause missing normalize_identifier

**What goes wrong:** `_compile_predicate` currently calls `q(f)` where `f` is the raw Python field name from the predicate. After introducing normalization, `q("order_id")` produces `"order_id"` but Snowflake stores `ORDER_ID`. This would produce a working query (Snowflake is case-insensitive for unquoted identifiers accessed via `"order_id"`) but breaks the semantic contract and may fail for quoted-lowercase columns.

**Why it happens:** The Lookup predicate only stores `field_name` (the Python name). The SQL builder must apply normalization when compiling predicates, just as it does for SELECT columns.

**How to avoid:** Update every `case Exact(field_name=f, value=v):` branch in `_compile_predicate` to normalize `f` before quoting: `normalized_f = self.dialect.normalize_identifier(f); return f"{q(normalized_f)} = {ph}", [v]`.

**Warning signs:** Tests that construct predicates with uppercase field names but assert on lowercase SQL output.

### Pitfall 2: `source=` leaking into `__repr__` unexpectedly

**What goes wrong:** If `source` is added to `Field.__repr__()` unconditionally, existing tests that snapshot `repr(Sales.revenue)` will break.

**How to avoid:** Show `source=` in repr only when set. Decision deferred to Claude per CONTEXT.md. Recommended: `Field('revenue')` when source is None, `Field('revenue', source='REVENUE')` when source is set.

### Pitfall 3: basedpyright `__get__` overload

**What goes wrong:** basedpyright strict mode can reject the `__get__` implementation if the implementation signature is too broad or doesn't match both overloads. Common failure: `reportReturnType` when returning `self` (typed as `Field[T]`) from a branch typed as `T`.

**How to avoid:** The implementation `-> Field[T] | T` satisfies both overloads. Verify with `uv run basedpyright` after each change. As of Python 3.14 + typing module, the overload pattern verified to work (see Code Examples).

### Pitfall 4: Jinja2 template — None vs "TODO:" data_type

**What goes wrong:** Currently the template does not use `data_type` at all — it emits bare `Metric()`. After this phase, `data_type` drives the `[T]` subscript. If `data_type` is `None` (for GEOGRAPHY etc.), the template must not emit `Metric[None]()`.

**How to avoid:** `_build_model_context()` in `python_renderer.py` should translate `None` data_type to `"Any"` (adding `needs_any = True` flag), and store the TODO comment separately. The template then always emits `Metric[{{ field.data_type }}]()` where `data_type` is always a non-None string.

### Pitfall 5: Existing test assertions on `Metric()`

**What goes wrong:** Existing tests in `test_python_renderer.py` and `test_cli.py` assert `"revenue = Metric()" in source`. After this change, the output will be `revenue = Metric[int]()`, breaking these assertions.

**How to avoid:** These tests must be updated to assert the new form. This is a known required change, not a regression. The plan should explicitly list these tests for update.

Affected assertions (verified from test file inspection):
- `test_python_renderer.py`: 5+ assertions checking `"= Metric()"`, `"= Dimension()"`, `"= Fact()"`
- `test_cli.py`: 4+ assertions checking `"revenue = Metric()"`, `"country = Dimension()"`, etc.

### Pitfall 6: `needs_any` import logic

**What goes wrong:** If any field has `data_type=None` (mapped to `Any`) but the renderer doesn't add `from typing import Any` to the output, the generated code is invalid Python.

**How to avoid:** Add `needs_any` flag to `render_views()` computed alongside `needs_datetime`, and add it to the template.

### Generic Field with source= parameter

```python
# Source: verified via Python 3.14 runtime testing
from __future__ import annotations
from typing import TypeVar, Generic, overload

T = TypeVar("T")

class Field(Generic[T]):
    def __init__(self, source: str | None = None) -> None:
        self.name: str | None = None
        self.source: str | None = source
        self.owner: type | None = None

    def __set_name__(self, owner: type, name: str) -> None:
        self.name = name
        self.owner = owner

    @overload
    def __get__(self, obj: None, cls: type) -> Field[T]: ...
    @overload
    def __get__(self, obj: object, cls: type) -> T: ...
    def __get__(self, obj: object | None, cls: type) -> Field[T] | T:
        if obj is None:
            return self
        raise AttributeError(
            f"Field '{self.name}' cannot be accessed from instances. "
            f"Use {cls.__name__}.{self.name} instead."
        )

    def __repr__(self) -> str:
        cls_name = type(self).__name__
        if self.name is None:
            return f"{cls_name}(unbound)"
        if self.source is not None:
            return f"{cls_name}('{self.name}', source='{self.source}')"
        return f"{cls_name}('{self.name}')"

    __hash__ = object.__hash__  # preserve hash despite __eq__ override


class Metric(Field[T]):
    pass  # inherits all Field behaviour including Generic subscript


class Dimension(Field[T]):
    pass


class Fact(Field[T]):
    pass
```

### normalize_identifier on Dialect

```python
# In src/cubano/engines/sql.py
from abc import ABC, abstractmethod

class Dialect(ABC):
    @abstractmethod
    def normalize_identifier(self, name: str) -> str:
        """
        Fold a Python field name to the SQL column name the warehouse stores.

        Called when field.source is None. Snowflake stores unquoted identifiers
        as UPPERCASE; Databricks stores them as lowercase; Mock passes through.
        """
        pass

    @abstractmethod
    def quote_identifier(self, name: str) -> str: ...

    @abstractmethod
    def wrap_metric(self, field_name: str) -> str: ...


class SnowflakeDialect(Dialect):
    def normalize_identifier(self, name: str) -> str:
        return name.upper()

    def quote_identifier(self, name: str) -> str:
        escaped = name.replace('"', '""')
        return f'"{escaped}"'

    def wrap_metric(self, field_name: str) -> str:
        return f"AGG({self.quote_identifier(field_name)})"


class DatabricksDialect(Dialect):
    def normalize_identifier(self, name: str) -> str:
        return name.lower()

    # ... rest unchanged

class MockDialect(Dialect):
    def normalize_identifier(self, name: str) -> str:
        return name  # identity — no folding in tests

    # ... rest unchanged
```

### SQL builder column resolution

```python
# In SQLBuilder._build_select_clause (updated)
def _resolve_col_name(self, field: Any) -> str:
    """Resolve SQL-facing column name for a field descriptor."""
    assert field.name is not None
    if field.source is not None:
        return field.source
    return self.dialect.normalize_identifier(field.name)

def _build_select_clause(self, query: Any) -> str:
    select_items: list[str] = []
    for metric in query._metrics:
        col_name = self._resolve_col_name(metric)
        wrapped = self.dialect.wrap_metric(col_name)
        select_items.append(wrapped)
    for dim in query._dimensions:
        col_name = self._resolve_col_name(dim)
        quoted = self.dialect.quote_identifier(col_name)
        select_items.append(quoted)
    return "SELECT " + ", ".join(select_items)
```

## Open Questions

1. **WHERE clause `source` lookup for field_name in predicates**
   - What we know: Predicates store the Python `field_name` string (from `field.name` at predicate creation time). The SQL builder compiles predicates independently of the field descriptor object.
   - What's unclear: `source=` overrides are on the Field descriptor. If a user defines `order_id = Metric[int](source="ORDER_ID")` on a Snowflake model with normalize=uppercase, the predicate created by `Sales.order_id == 5` stores `field_name="order_id"`. The SQL builder should emit `"ORDER_ID" = %s` (normalized), not `"order_id" = %s`.
   - Recommendation: Apply `normalize_identifier(f)` in `_compile_predicate` for all Lookup branches. This is sufficient because: (a) the normalize step handles default-casing columns correctly, and (b) explicitly-quoted columns (e.g., `source="order_id"` on Snowflake) are not reachable via predicates — predicates don't carry `source`. Documenting this limitation (or deferring source-aware WHERE to v0.3) is acceptable.

2. **`source=` in `__repr__` — show or hide?**
   - What we know: CONTEXT.md marks this as Claude's discretion.
   - What's unclear: Whether users will see reprs in debugging and be confused by the extra field.
   - Recommendation: Show `source=` in repr when set (`Field('order_id', source='ORDER_ID')`). This aids debugging and is consistent with the Django `db_column=` convention where showing the mapping is helpful.

3. **Typer exit code 2 collision documentation**
   - What we know: Typer fires code 2 for missing `--backend`; our code 2 is "invalid backend value" (value provided but unrecognized). These are different invocations.
   - What's unclear: Whether to document the distinction in the `--help` text or just in the how-to guide.
   - Recommendation: Document in both places. In `--help`, note "Exit code 2 is also used by the CLI parser for missing required arguments." In the how-to, show the full table with this note.

4. **Exception subclasses for view-not-found vs connection-error**
   - What we know: Currently `introspect()` raises only `RuntimeError` for all failures.
   - What's unclear: Whether to introduce `CubanoViewNotFoundError`/`CubanoConnectionError` (clean library design) or message-pattern matching (simpler).
   - Recommendation: Introduce the two exception subclasses in `cubano.engines.base`. This is a small addition, keeps the library clean, and enables the distinct exit codes without fragile string matching.
