  <action>
Create `.github/workflows/docs.yml` using the two-job pattern (build then deploy). This is the GitHub-recommended modern approach — it uses `actions/upload-pages-artifact` and `actions/deploy-pages` rather than `mkdocs gh-deploy`, avoiding the need for `contents: write` permission.

```yaml
name: Docs

on:
  push:
    branches:
      - main
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: docs-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    name: Build docs
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up uv
        uses: astral-sh/setup-uv@v7
        with:
          enable-cache: true
          cache-dependency-glob: "uv.lock"

      - name: Sync docs dependencies
        run: uv sync --locked --group docs

      - name: Build docs
        run: uv run mkdocs build --strict
        env:
          DOCSEARCH_APP_ID: ${{ secrets.DOCSEARCH_APP_ID }}
          DOCSEARCH_API_KEY: ${{ secrets.DOCSEARCH_API_KEY }}

      - name: Upload pages artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: site/

  deploy:
    name: Deploy to GitHub Pages
    needs: build
    runs-on: ubuntu-latest
    timeout-minutes: 5
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**Key design decisions (from research):**
- `permissions: pages: write` and `id-token: write` are required for `actions/deploy-pages` — do NOT add `contents: write` (not needed with this pattern)
- `concurrency: cancel-in-progress: true` prevents multiple simultaneous deployments on rapid pushes to main
- `fetch-depth: 0` is needed if `mkdocs-git-revision-date-localized` plugin is used; harmless if not
- `workflow_dispatch` allows manual trigger from GitHub Actions UI without a push
- The `environment: github-pages` block links to the GitHub Pages environment and outputs the deployment URL
- `uv sync --locked --group docs` installs only the docs dependencies, NOT the dev group — keeps the build lean

**Note on GitHub Pages setup:** GitHub Pages must be configured in the repository settings to use "GitHub Actions" as the source (not the legacy branch-based deployment). This is a one-time manual repo configuration. Add a note about this in the workflow file comment:

```yaml
# SETUP REQUIRED: In GitHub repo Settings > Pages > Source, select "GitHub Actions"
# This enables the actions/deploy-pages action to publish the site.
```

**Note on DocSearch secrets:** The `env:` block on the "Build docs" step injects `DOCSEARCH_APP_ID` and `DOCSEARCH_API_KEY` from GitHub repository secrets into the `mkdocs build` environment. MkDocs Material reads these via `!ENV` directives in `mkdocs.yml` (configured in plan 10-01 Task 2). Store these secrets in the GitHub repo under Settings > Secrets and variables > Actions. Obtain the credentials from https://docsearch.algolia.com (free for open source). If the secrets are not yet set, `mkdocs build --strict` will still succeed — Material theme degrades gracefully when Algolia credentials are absent, falling back to the standard search plugin.
