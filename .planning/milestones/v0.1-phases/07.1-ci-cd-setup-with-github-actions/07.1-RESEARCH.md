## Summary

GitHub Actions provides mature, well-integrated CI/CD capabilities for Python packages. The ecosystem has standardized around Trusted Publishing (OIDC) for PyPI releases, eliminating API token management. The `astral-sh/setup-uv` action and `pypa/gh-action-pypi-publish` action are the official, blessed solutions for this workflow.

For Cubano's specific needs (uv-based build, strict quality gates, semver tagging), the implementation is straightforward: multiple workflows handling CI (quality checks), PR validation (status checks + coverage), and release (build + validate + publish). The uv toolchain has first-class GitHub Actions support with built-in caching, Python version matrices, and isolated test environments.

Key decision points center on changelog generation (git-cliff vs commitizen), coverage reporting format, and concurrency management for PR checks. All technical requirements from the phase context are well-supported by the current ecosystem.

**Primary recommendation:** Use separate workflows for CI (`.github/workflows/ci.yml`), PR checks (`.github/workflows/pr.yml`), and release (`.github/workflows/release.yml`) to maintain clear separation of concerns and permissions. Leverage Trusted Publishing for PyPI, enable built-in caching in setup-uv, and use git-cliff for changelog generation (Rust-based, fast, highly customizable).

## GitHub Actions Workflow Architecture

### Workflow Organization

**Best Practice:** Use separate workflow files for different concerns to maintain clean permissions boundaries and clear responsibilities.

**Recommended structure:**
```
.github/workflows/
├── ci.yml           # Quality checks on all branches (typecheck, lint, format, tests)
├── pr.yml           # PR-specific actions (coverage reports)
└── release.yml      # Build, validate, publish to PyPI on tags
```

**Why separate files:**
- **Security:** Release workflow needs `id-token: write` for Trusted Publishing; CI workflows should have minimal permissions
- **Performance:** PR checks can use `cancel-in-progress: true` to cancel outdated runs; releases should never be canceled
- **Clarity:** Each workflow has a single responsibility, making debugging and modification easier

### Concurrency Control

GitHub Actions `concurrency` key allows controlling workflow execution:

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true  # Cancel old runs when new commits pushed
```

**Use cases:**
- **CI/PR workflows:** Set `cancel-in-progress: true` to avoid wasting resources on outdated commits
- **Release workflow:** Omit concurrency control or set `cancel-in-progress: false` to prevent accidental cancellations

**Source:** [GitHub Docs - Control concurrency](https://docs.github.com/actions/writing-workflows/choosing-what-your-workflow-does/control-the-concurrency-of-workflows-and-jobs)

### Matrix Strategy for Python Versions

Use matrix testing to validate against multiple Python versions:

```yaml
strategy:
  matrix:
    python-version: ["3.11", "3.14"]
steps:
  - uses: astral-sh/setup-uv@v7
    with:
      python-version: ${{ matrix.python-version }}
  - run: uv sync --locked --all-extras --dev
  - run: uv run pytest tests
```

**Key points:**
- Matrix jobs run in parallel by default
- `astral-sh/setup-uv` overrides Python version from `pyproject.toml` when `python-version` is specified
- Use `--locked` flag with `uv sync` for reproducible builds in CI

**Source:** [uv Docs - GitHub Actions](https://docs.astral.sh/uv/guides/integration/github/)

### Job Dependencies and Artifacts

For release workflows, use job dependencies to sequence build → validate → publish:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: uv build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

  validate:
    needs: build  # Waits for build to complete
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - run: # validation commands

  publish:
    needs: validate  # Waits for validation to complete
    runs-on: ubuntu-latest
    steps:
      - uses: pypa/gh-action-pypi-publish@release/v1
```

**Benefits:**
- Clear linear flow: build once, validate, then publish
- Build artifacts are isolated and reused across jobs
- Failed validation prevents publishing (hard block as required)

**Source:** [GitHub Actions - Publishing Python packages](https://packaging.python.org/en/latest/guides/publishing-package-distribution-releases-using-github-actions-ci-cd-workflows/)

## Quality Check Implementation

### TypeCheck: basedpyright with Strict Mode

**Current configuration (pyproject.toml):**
```toml
[tool.basedpyright]
pythonVersion = "3.11"
typeCheckingMode = "strict"
include = ["src", "tests"]
reportPrivateUsage = false
```

**GitHub Actions integration:**
```yaml
- name: Type check
  run: uv run basedpyright
```

**Key advantage:** basedpyright automatically detects GitHub Actions environment and formats output as workflow commands. Errors appear inline on affected code lines in PRs without additional configuration.

**No action wrapper needed:** Unlike pyright (requires third-party actions with boilerplate), basedpyright works directly with zero setup.

**Source:** [basedpyright Docs - CI Integration](https://docs.basedpyright.com/v1.18.2/benefits-over-pyright/improved-ci-integration/)

**Current configuration (pyproject.toml):**

```toml
[tool.ruff]
target-version = "py311"
line-length = 100
src = ["src"]

[tool.ruff.lint]
```

**GitHub Actions integration:**

**Key flags:**

### Required Status Checks Configuration

**Branch protection setup:**
1. Navigate to repository Settings → Branches → Branch protection rules
2. Add rule for `main` branch
3. Enable "Require status checks to pass before merging"
4. Search for check names: `typecheck`, `lint`, `format`, `test`
5. Enable "Require branches to be up to date before merging" (strict mode) or leave disabled (loose mode)

**Critical requirement:** Job names must be UNIQUE across all workflows. Duplicate job names cause ambiguous status checks and block merges.

**Check visibility:** GitHub only shows checks that have run on the protected branch within the last 7 days. To make checks appear:
1. Add `push: branches: [main]` trigger to CI workflow
2. Push to main to run checks once
3. Refresh branch protection page
4. Check names will now appear in the selection list

**Admin override:** Admins can bypass branch protection by default. This can be changed in repository settings.

**Source:** [GitHub Docs - About protected branches](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches)

### Security: Dependabot Configuration

**Configuration file:** `.github/dependabot.yml`

**Basic setup for Python with uv:**
```yaml
version: 2
updates:
  - package-ecosystem: "pip"  # Use "pip" for pip, pipenv, poetry, uv
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    groups:
      dev-dependencies:
        dependency-type: "development"
      production-dependencies:
        dependency-type: "production"
```

**Key points:**
- **Ecosystem:** Use `"pip"` for all Python package managers (pip, pipenv, poetry, uv)
- **Grouping:** Combine related dependencies into single PRs for easier review
- **Frequency:** Weekly is recommended for active projects; monthly for stable projects
- **Security updates:** Automatically enabled; run separately from version updates with 10-PR limit

**For Cubano (zero runtime dependencies):**
```yaml

  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
```

## Release & Publication Pipeline

### Trusted Publishing (OIDC) with PyPI

**What it is:** OpenID Connect-based authentication that exchanges short-lived identity tokens between GitHub Actions and PyPI, eliminating manual API token management.

**Benefits:**
- **Security:** Tokens expire automatically (15 minutes), no long-lived credentials
- **Usability:** No copy-paste of API tokens into secrets
- **Automation:** One-time PyPI configuration, then fully automated releases

**Setup steps:**

1. **PyPI configuration** (one-time, manual):
   - Sign in to PyPI
   - Navigate to project settings → Publishing → Add publisher
   - Configure GitHub Actions publisher:
     - Repository owner: `anentropic` (or your GitHub username)
     - Repository name: `cubano`
     - Workflow filename: `release.yml`
     - Environment name: `pypi` (optional but recommended)

2. **Workflow configuration:**
```yaml
jobs:
  publish:
    runs-on: ubuntu-latest
    environment:
      name: pypi
      url: https://pypi.org/p/cubano
    permissions:
      id-token: write  # CRITICAL: Required for OIDC token
    steps:
      - uses: pypa/gh-action-pypi-publish@release/v1
        # No username/password needed!
```

**Security model:**
- Trust is scoped to specific repository + workflow + environment
- Cannot be triggered from forks or different workflows
- Admin-only release tags prevent unauthorized publishes

**Source:** [PyPI Docs - Trusted Publishing](https://docs.pypi.org/trusted-publishers/)

### Changelog Generation from Conventional Commits

**Conventional Commits specification:**
- `fix:` commits → PATCH version bump (1.0.0 → 1.0.1)
- `feat:` commits → MINOR version bump (1.0.0 → 1.1.0)
- `BREAKING CHANGE:` footer or `!` suffix → MAJOR version bump (1.0.0 → 2.0.0)

**Source:** [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)

**Changelog generation tools:**

**Option 1: git-cliff (RECOMMENDED)**
- Written in Rust (fast, single binary)
- Highly customizable via configuration file
- Generates changelog from git history
- Language-agnostic (works with any git repo)
- Installation: `cargo install git-cliff` or download binary

**Example workflow:**
```yaml
- name: Generate changelog
  run: |
    curl -L https://github.com/orhun/git-cliff/releases/latest/download/git-cliff-linux-amd64.tar.gz | tar xz
    ./git-cliff-*/git-cliff --latest --output CHANGELOG.md
```

**Option 2: commitizen**
- Python-native tool
- Enforces conventional commits
- Auto-bumps versions and generates changelogs
- Requires installation: `uv pip install commitizen`

**Example workflow:**
```yaml
- name: Generate changelog
  run: |
    uv run commitizen changelog
```

**Recommendation:** Use git-cliff for simpler integration (no Python dependency, single binary, faster). Commitizen is better if you want to enforce commit message format during development (pre-commit hooks).

**Sources:**
- [git-cliff](https://git-cliff.org/)
- [commitizen](https://commitizen-tools.github.io/commitizen/)

### PEP 740 Attestations

**What it is:** Digital signatures for distribution files, providing cryptographic proof of origin.

**Implementation:** `pypa/gh-action-pypi-publish@v1.11.0+` automatically generates and uploads attestations using Sigstore.

**No action required:** Attestations are enabled by default. Both authentication and attestations use the same GitHub-backed OIDC identity.

**Disable if needed:**
```yaml
- uses: pypa/gh-action-pypi-publish@release/v1
  with:
    attestations: false
```

**Source:** [pypa/gh-action-pypi-publish](https://github.com/pypa/gh-action-pypi-publish)

### Release Workflow Trigger

**Tag-based releases:**
```yaml
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'  # Matches v1.0.0, v2.3.4, etc.
      - 'v[0-9]+.[0-9]+.[0-9]+-*' # Matches v1.0.0-beta.1, etc.
```

**Extract version from tag:**
```yaml
- name: Get version from tag
  id: get_version
  run: echo "VERSION=${GITHUB_REF#refs/tags/v}" >> $GITHUB_OUTPUT

- name: Update version in pyproject.toml
  run: |
    sed -i 's/version = ".*"/version = "${{ steps.get_version.outputs.VERSION }}"/' pyproject.toml
```

**Admin-only restriction:** Configure branch protection or use GitHub Environments to restrict who can push tags.

## Best Practices & Recommendations

### 1. Caching Strategy

**Use built-in caching in setup-uv:**
```yaml
- uses: astral-sh/setup-uv@v7
  with:
    enable-cache: true
    cache-dependency-glob: "uv.lock"
```

**Cache pruning:** Run `uv cache prune --ci` at the end of workflows to reduce cache size:
```yaml
- name: Prune cache
  if: always()
  run: uv cache prune --ci
```

**Benefits:**
- Faster CI runs (dependencies cached between runs)
- Automatic cache invalidation when `uv.lock` changes
- Optimized cache size for CI (prune removes unused artifacts)

**Source:** [uv Docs - GitHub Actions](https://docs.astral.sh/uv/guides/integration/github/)

### 2. Permissions Principle of Least Privilege

**Default permissions:** Set to read-only globally, elevate per-job as needed:

```yaml
permissions:
  contents: read

jobs:
  test:
    runs-on: ubuntu-latest
    # Inherits read-only permissions

  coverage:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write  # Only for coverage comments
      contents: read

  publish:
    runs-on: ubuntu-latest
    permissions:
      id-token: write  # Only for Trusted Publishing
      contents: read
```

**Source:** [GitHub Actions Best Practices](https://docs.github.com/en/actions/security-guides/security-hardening-for-github-actions)

### 3. Workflow Separation

**Separate workflows by concern:**
- `ci.yml`: Quality checks (typecheck, lint, format, tests) on all pushes
- `pr.yml`: PR-specific actions (coverage reports) on pull_request events
- `release.yml`: Build, validate, publish on tag pushes

**Benefits:**
- Clear permission boundaries
- Easier to debug and modify
- Reduced complexity per file
- Better concurrency control

### 4. Job Names and Status Checks

**Use descriptive, unique job names:**
```yaml
jobs:
  typecheck:  # Status check name: "typecheck"
    name: Type check with basedpyright
    runs-on: ubuntu-latest
    steps: ...

  lint:  # Status check name: "lint"
    name: Lint with ruff
    runs-on: ubuntu-latest
    steps: ...
```

**Avoid duplicate job names across workflows:** GitHub uses job names for status checks. Duplicates cause ambiguous results and block merges.

### 5. Matrix Testing Strategy

**Test minimum and maximum supported versions:**
```yaml
strategy:
  matrix:
    python-version: ["3.11", "3.14"]  # Min and max
  fail-fast: false  # Continue testing other versions if one fails
```

**Benefits:**
- Ensures compatibility with min supported version (3.11)
- Catches issues with latest Python features (3.14)
- `fail-fast: false` provides complete test results across versions

### 6. Concurrency Management

**CI workflows (cancel outdated runs):**
```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

**Release workflows (never cancel):**
```yaml
# No concurrency key, or:
concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false
```

### 7. Timeouts

**Set reasonable timeouts to prevent hung jobs:**
```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps: ...
```

**Recommended timeouts:**
- Type checking: 5 minutes
- Linting: 5 minutes
- Tests: 10 minutes
- Build: 10 minutes
- Validation: 10 minutes per Python version
- Publish: 5 minutes

### 8. Artifact Retention

**Control artifact retention for builds:**
```yaml
- uses: actions/upload-artifact@v4
  with:
    name: dist
    path: dist/
    retention-days: 7  # Delete after 7 days to save storage
```

**For release artifacts:** Download from GitHub Releases or PyPI instead of Actions artifacts.

### 9. Environment Protection Rules

**Configure PyPI environment for release safety:**
1. Navigate to Settings → Environments → New environment
2. Create environment named `pypi`
3. Configure protection rules:
   - Required reviewers: (optional) require manual approval
   - Deployment branches: Only allow `main` branch and tags matching `v*`

**Benefits:**
- Extra safety layer before PyPI publish
- Audit trail of releases
- Scoped secrets (PYPI_TOKEN if using token auth instead of Trusted Publishing)

### 10. Testing Against Built Distributions

**Test installed package, not source tree:**
```yaml
- name: Build package
  run: uv build

- name: Test wheel installation
  run: |
    uv venv test-env
    uv pip install --python test-env/bin/python dist/*.whl
    test-env/bin/python -c "import cubano; assert cubano.__version__"
```

**Why:** Increases test fidelity to user experience. Catches packaging errors (missing files, incorrect entry points, broken imports).

**Source:** [GitHub Actions - Testing Python packages](https://learn.scientific-python.org/development/guides/gha-pure/)

### Key Findings

1. **Trusted Publishing is the standard:** PyPI's OIDC implementation eliminates API token management. Setup is one-time manual configuration on PyPI, then fully automated via `pypa/gh-action-pypi-publish@release/v1` with `id-token: write` permission.

2. **uv has first-class GitHub Actions support:** Official `astral-sh/setup-uv@v7` action provides built-in caching, Python version matrix support, and isolated environment testing. Use `uv build` for building, `uv run` for executing tools, and `uv sync --locked` for reproducible installs.

3. **basedpyright requires zero configuration:** Automatically detects GitHub Actions and formats output as workflow commands. Direct CLI invocation (`uv run basedpyright`) is all that's needed.

4. **Separate workflows for separation of concerns:** CI (quality checks), PR (coverage reports), Release (build/validate/publish) should be separate files with distinct permissions and concurrency strategies.

5. **Changelog automation is mature:** Both git-cliff (Rust, fast) and commitizen (Python) support conventional commits. git-cliff recommended for simpler integration (single binary, no dependencies).
