# Phase 07.1: CI/CD Setup with GitHub Actions - Context

**Gathered:** 2026-02-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Set up GitHub Actions workflows for automated testing, quality validation, and PyPI release. Covers CI triggers and gates on all commits, PR requirements for main, and automated release pipelines with validation before publishing to PyPI. Does NOT include cloud infrastructure, deployment to production, or monitoring systems.

</domain>

<decisions>
## Implementation Decisions

### Workflow Triggers & Branch Strategy

- **CI execution:** Run on all branches (every push triggers CI)
- **Main branch protection:** PR merge requires passing all quality checks; admins can override
- **Release trigger:** Auto-publish to PyPI on git tags matching semver (v1.2.3)
- **Trunk-based releasing:** Use main branch only; no develop branch
- **Hotfix strategy:** Create hotfix branches from release tags, merge back to main with new patch tag
- **PR notifications:** GitHub checks only (no Slack/email alerts)

### Quality Check Scope

- **Required checks:** All 4 must pass before merge: typecheck (strict), lint, format, tests
- **Typecheck mode:** Strict (basedpyright strict=true as configured)
- **Coverage tracking:** Measure and report in PRs, but don't enforce minimum threshold
- **Security scanning:** Use GitHub Dependabot for dependency vulnerability scanning (no additional security tools)

### Python Versions & Release Automation

- **Test matrix:** Python 3.11 (minimum supported) and 3.14 (current development target)
- **Changelog:** Auto-generate on release using conventional commits
- **Release authorization:** Admins only can push release tags and trigger publishing
- **Release validation:** Build and test wheel/sdist before uploading to PyPI

### Build Validation

- **Validation scope:** Install + import tests AND wheel integrity checks (metadata, py.typed marker, structure)
- **Isolation:** Run validation in isolated virtual environments for each Python version (3.11 and 3.14 separately)
- **Validation blocking:** Hard block — failed validation prevents PyPI upload (no override path)

### Claude's Discretion

- Exact GitHub Actions workflow file structure and step sequencing
- Caching strategy for dependencies and build artifacts
- Specific test timeouts and concurrency settings
- Changelog formatting and entry selection from commits
- Dependabot configuration details (update frequency, grouping)

</decisions>

<specifics>
## Specific Ideas

- Workflows should follow GitHub's best practices for Python packages (e.g., matrix strategy, conditional steps)
- Consider using standard actions: actions/checkout, actions/setup-python, pypa/build, etc.
- Tag format should follow semantic versioning (v0.1.0, v1.0.0-beta.1, etc.)

</specifics>

<deferred>
## Deferred Ideas

- Automated canary releases (e.g., dev/pre-release workflows)
- Integration tests against real Snowflake/Databricks instances in CI
- Automated hotline/emergency release process (outside phase scope)
- GitHub status badges in README (documentation task, not CI/CD setup)

</deferred>
