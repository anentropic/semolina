<decisions_made>

- **CLI Framework**: Typer over Click/argparse
  - Reason: Type-hint based, modern, excellent testing support via CliRunner, integrates seamlessly with Rich

- **SQL Templating**: Jinja2 with whitespace control markers ({%- and -%})
  - Reason: Prevents whitespace explosion in generated SQL, templates in separate .jinja2 files for maintainability

- **Error Handling**: ExceptionGroup (Python 3.11+) for multi-error reporting
  - Reason: Native language feature, no custom error collection patterns needed, users see all issues at once

- **stdout/stderr Separation**: Rich Console instances for SQL (stdout) vs diagnostics (stderr)
  - Reason: Enables piping; SQL output unpolluted by progress/status messages

- **Module Loading**: importlib.util for dynamic module loading from arbitrary paths
  - Reason: Allows users to specify model files anywhere without sys.path manipulation

- **Wave Assignment**: Wave 1 (CLI foundation only), Waves 2 (templates + introspection parallel), Wave 3 (integration)
  - Reason: Enables parallelization; Wave 2 can start before Wave 1 completes; Wave 3 waits for both

- **CODEGEN-05 Interpretation**: Validate Python input syntax (ast.parse), NOT SQL output
  - Reason: Per user decision ("trust template correctness"), but validate input to catch user mistakes early

- **CODEGEN-06 Responsibility**: Delegated to Cubano model API at class definition time
  - Reason: SemanticView.__init_subclass__ prevents invalid circular dependencies; codegen doesn't re-validate

</decisions_made>

## Phase 9 Overview

Build a CLI tool (`cubano codegen`) that generates warehouse-native SQL from Cubano Python model definitions. Users specify model files and target backend; tool outputs CREATE SEMANTIC VIEW (Snowflake) or metric view definitions (Databricks).

## Requirements Addressed

All 8 CODEGEN requirements:
- CODEGEN-01: CLI executable and user-friendly
- CODEGEN-02: Snowflake CREATE SEMANTIC VIEW with AGG() wrapping
- CODEGEN-03: Databricks metric view definitions with MEASURE()
- CODEGEN-04: Process all models in a file
- CODEGEN-05: Validate Python input syntax before processing
- CODEGEN-06: Circular relationship validation (delegated to model API)
- CODEGEN-07: --backend, --output, flexible input (files/dirs/globs)
- CODEGEN-08: SQL formatted for readability, docstring comments

## Plan Architecture

**Wave 1 (CLI Foundation, 09-01):**
- Typer app with `cubano codegen` command

- Help text, version command, error handling
- Sets up stubs for template & introspection logic

**Wave 2A (SQL Templates, 09-02):**
- Jinja2 templates for Snowflake and Databricks
