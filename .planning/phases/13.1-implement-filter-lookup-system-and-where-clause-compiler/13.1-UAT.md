## Tests

### 1. Field comparison operators produce Predicate nodes
expected: In a Python REPL or test, `Sales.revenue > 1000` returns a `Gt` predicate node (not a Q object). Similarly `==` returns `Exact`, `!=` returns `Not(Exact(...))`, `<` returns `Lt`, `<=` returns `Lte`, `>=` returns `Gte`. Each node has `.field_name` and `.value` attributes.
result: issue
reported: "!= produces Not(Exact(...)) which compiles to NOT (a = b) instead of a != b. Should be a distinct operator (NotEqual/Ne) so generated SQL looks natural to users. Name the bounded scalar TypeVar 'ScalarT'."
severity: minor

### 2. Named filter methods work on Field descriptors
expected: `Sales.country.in_(["US", "CA"])` returns `In(field_name="country", value=["US", "CA"])`. Similarly `.between(lo, hi)`, `.like(pattern)`, `.ilike(pattern)`, `.startswith(prefix)`, `.endswith(suffix)`, `.iexact(value)`, `.isnull()` all return the corresponding Predicate node types.
result: issue
reported: "Lookup generic type args are too loose. In should be Sequence or Collection (minimum for compiler usage), Between should be tuple[Any, Any], scalar lookups (Exact, Gt, Gte, Lt, Lte) should use a bounded TypeVar named ScalarT — not bare Any."
severity: minor

### 3. Predicate composition with &, |, ~ operators
expected: `(Sales.country == "US") & (Sales.revenue > 1000)` produces `And(left=Exact(...), right=Gt(...))`. Using `|` produces `Or(...)`. Using `~` produces `Not(...)`. These compose arbitrarily: `(a & b) | ~c` works.
result: pass

### 4. where() accepts varargs and filters None
expected: `Model.query().where(Sales.country == "US", Sales.revenue > 1000)` ANDs both conditions. `Model.query().where(None)` is a no-op (returns unchanged query). Multiple `.where()` calls chain with AND.
result: skipped
reason: Functional correctness — covered by unit tests

### 5. to_sql() renders real WHERE clauses with inline values
expected: `Sales.query().where(Sales.country == "US").to_sql()` returns SQL string containing `WHERE "country" = 'US'` (values rendered inline via repr). No more `WHERE 1=1` placeholder.
result: pass

### 6. Parameterized SQL via build_select_with_params()
expected: Calling `build_select_with_params()` on a query with filters returns a tuple `(sql_template, params)` where the SQL contains `%s` placeholders (Snowflake/Mock) and `params` is a list of bind values. No literal values in the SQL template.
result: pass

### 7. Public API exports Predicate (Q removed)
expected: `from cubano import Predicate` succeeds. `from cubano import Q` raises ImportError. `Predicate` is in `cubano.__all__`.
result: pass
