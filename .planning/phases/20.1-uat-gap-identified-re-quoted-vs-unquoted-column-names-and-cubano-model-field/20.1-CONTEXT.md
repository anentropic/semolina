# Phase 20.1: UAT Gap â€” Quoted vs Unquoted Column Names and Cubano Model Field - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two UAT gaps discovered during live Snowflake testing of Phase 20's reverse codegen CLI:

1. **Column name case mismatch**: Snowflake returns uppercase column names (e.g., `ORDER_ID`) from
   `SHOW COLUMNS IN VIEW`, but Python convention expects lowercase snake_case field names. The
   `Field` descriptors have no mechanism to carry a separate SQL column name, so generated code
   would either be un-Pythonic or produce broken SQL (quoted lowercase â‰  uppercase stored column).

2. **Generic type support on Field**: The codegen type-mapping pipeline (`type_map.py`) already
   maps SQL types to Python strings (e.g., `FIXED â†’ int`), but the generated code emits bare
   `order_id = Metric()` with no type information. Making this type-safe requires `Field` to be
   `Generic[T]` so `Metric[int]()` is a valid, type-checker-friendly expression.

3. **Distinct exit codes on the codegen CLI**: All errors currently exit with code 1. Distinct
   codes per error category are needed for good CLI practice.

This phase does NOT add new CLI features, new warehouse backends, or changes to the query engine
beyond what's needed to support `source=` identifier bridging.

</domain>

<decisions>
## Implementation Decisions

### Column name mapping strategy

- Python field name = `original_column_name.lower()` (e.g., `ORDER_ID` â†’ `order_id`)
- Add `source: str | None = None` parameter to `Field.__init__()` (and therefore to
  `Metric`, `Dimension`, `Fact` which inherit it).
- `source=None` (default): the SQL builder defers to `dialect.normalize_identifier(field.name)`
  to get the SQL-facing column name. Each dialect knows its identifier folding:
  - `SnowflakeDialect.normalize_identifier(name)` â†’ `name.upper()`
  - `DatabricksDialect.normalize_identifier(name)` â†’ `name.lower()`
  - `MockDialect.normalize_identifier(name)` â†’ identity (name unchanged)
- `source="some_value"` (explicit): the SQL builder uses that string verbatim (after quoting).
  This is how users specify explicitly-quoted, non-default-casing columns:
  - Snowflake quoted-lowercase column `"order_id"` â†’ `source="order_id"` (verbatim, not uppercased)

### When codegen emits source=

Codegen should emit `source=` only when it is necessary â€” i.e., when the dialect would
auto-normalize to a different name than the original:

- Rule: if `dialect.normalize_identifier(python_name) == original_column_name` â†’ no `source=`
- Otherwise â†’ emit `source=original_column_name`

Examples for Snowflake (normalize = uppercase):
- `ORDER_ID` â†’ python `order_id`, normalize(`order_id`) = `ORDER_ID` = original â†’ `order_id = Metric[int]()`
- `"order_id"` â†’ python `order_id`, normalize(`order_id`) = `ORDER_ID` â‰  `order_id` â†’ `order_id = Metric[int](source="order_id")`

This logic needs comprehensive test coverage and clear docs.

### SQL builder changes

- Each `Dialect` gains a `normalize_identifier(name: str) -> str` method alongside the
  existing `quote_identifier`.
- SQL builder resolves the SQL column name as:
  ```python
  col_name = field.source if field.source is not None else dialect.normalize_identifier(field.name)
  sql_ref = dialect.quote_identifier(col_name)
  ```
- No backward compatibility shim needed â€” the new behavior is the correct behavior. Existing
  hand-crafted models without `source=` will now get `normalize_identifier` applied (which is
  what Snowflake users already expected to happen implicitly).
- View names in `FROM` / `GROUP BY ALL`: left untouched. User supplies the view name in the
  correct case. We document that view names are always quoted verbatim, so Snowflake users
  with unquoted view names must use all-caps in their model's `view=` parameter.

### Generic[T] on Field

- `Field` becomes `Generic[T]` (using `typing.Generic`).
- `Metric`, `Dimension`, `Fact` inherit and expose `[T]` subscript via the generic parent.
- `__get__` overloads: return `Self` when `obj is None` (class access for query building),
  return `T` when `obj` is not None (instance access â€” currently unused in Cubano, but correct
  for type checkers).
- Generated code form: `order_id = Metric[int]()` â€” no explicit variable annotation; type
  checker infers `Metric[int]` from the descriptor.
- For unmapped/TODO SQL types: `order_id = Metric[Any]()  # TODO: GEOGRAPHY type`
- `import datetime` logic retained (drives `needs_datetime` for the template's `import datetime`).
- `from typing import Any` added to generated output when Any is used.

### Exit codes on the codegen CLI

- Motivation: good CLI convention (not driven by current scripting needs).
- Distinct codes:
  - `1` â€” Unexpected/unhandled error (catch-all)
  - `2` â€” Invalid backend (unknown or unsupported `--backend` value)
  - `3` â€” View not found in the warehouse
  - `4` â€” Connection / authentication failure
- Documentation: exit codes table added to **both** the `--help` text and the `codegen` how-to
  guide page.
- Note: Typer currently emits exit code 2 for its own validation errors (missing args, etc.).
  The plan should check whether to align or work around Typer's built-in exit code 2.

### Claude's Discretion

- `normalize_identifier` method signature and placement (on `Dialect` base class as abstract,
  or concrete on each subclass).
- Whether `source` should appear in `Field.__repr__()`.
- How to handle the Jinja2 template changes needed for the `[T]` subscript in `Metric[int]()`.
- basedpyright overload annotations for `__get__` â€” Claude to verify the correct form under
  strict mode.

</decisions>

<specifics>
## Specific Ideas

- The `source=` approach mirrors Django ORM's `db_column=` parameter on model fields â€” a
  familiar pattern for the target audience.
- The codegen output should remain copy-paste-ready: a user should be able to run `cubano codegen`
  and drop the result into their project without manual edits.
- Exit code Typer conflict: Typer itself uses exit code 2 for argument validation failures.
  If we also want 2 for "invalid backend", need to check if they collide or if Typer's
  validation fires before our logic.

</specifics>
