## Accomplishments

- Added `CubanoViewNotFoundError` and `CubanoConnectionError` to `cubano.engines.base` — both subclass `RuntimeError` for backward compatibility
- Wired typed exceptions into `SnowflakeEngine.introspect()` (ProgrammingError → view-not-found, DatabaseError → connection) and `DatabricksEngine.introspect()` (OperationalError → connection, DatabaseError → view-not-found)
- CLI now exits with distinct codes: 2 for invalid backend, 3 for view not found, 4 for connection failure, 1 for unexpected errors
- Exit codes table added to `cubano codegen --help` via `app.command(epilog=...)` in `__init__.py`

## Decisions Made

- `CubanoViewNotFoundError` and `CubanoConnectionError` both subclass `RuntimeError` — existing catch-all `except RuntimeError` callers continue to work without changes, while typed callers can now distinguish failure modes
- Epilog lives in `app.command(epilog=...)` in `__init__.py` rather than in the function docstring — keeps the docstring clean and avoids Typer stripping/mangling the text
- Snowflake `ProgrammingError` maps to view-not-found (SQL errors, invalid object names); `DatabaseError` maps to connection failure — aligns with Snowflake connector semantics
- Databricks `OperationalError` maps to connection failure (connector fires this for auth/network); `DatabaseError` maps to view-not-found (fires when DESCRIBE TABLE EXTENDED finds nothing)
