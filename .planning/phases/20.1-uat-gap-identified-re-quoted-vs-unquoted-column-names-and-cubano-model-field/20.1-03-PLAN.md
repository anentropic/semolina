    ```markdown
    ## Exit codes

    The `cubano codegen` command uses distinct exit codes so you can handle failure modes
    in scripts:

    | Exit code | Meaning |
    |-----------|---------|
    | 0 | Success — model class written to stdout |
    | 1 | Unexpected error (see stderr for details) |
    | 2 | Invalid `--backend` specifier — value provided but not recognised |
    | 3 | View not found — the warehouse has no semantic view with that name |
    | 4 | Connection failure — credentials missing or authentication rejected |

    > **Note:** Exit code 2 is also emitted by the CLI argument parser when `--backend` is
    > omitted entirely. Both cases mean "the backend could not be resolved."
    ```

    Also add the source= usage pattern for hand-crafted models (per CONTEXT.md specifics —
    this is within-phase docs scope, not deferred):

    Find the section that explains model field definitions (or add a new subsection). Add a
    brief explanation of when source= is needed:

    ```markdown
    ## Using source= for column name overrides

    By default, Cubano maps Python field names to SQL column names using the dialect's
    identifier folding (Snowflake uppercases unquoted identifiers; Databricks lowercases them).
    For a field `order_id`, Snowflake resolves `"ORDER_ID"` automatically.

    If your warehouse stores a column with non-default casing — for example, a quoted
    lowercase column `"order_id"` in Snowflake — you can override the SQL column name
    with `source=`:

    ~~~python
    class Orders(SemanticView, view="orders"):
        order_id = Metric[int](source="order_id")  # maps to quoted "order_id", not "ORDER_ID"
    ~~~

    The `cubano codegen` command emits `source=` automatically when introspection detects
    that a column uses non-default casing.
    ```
