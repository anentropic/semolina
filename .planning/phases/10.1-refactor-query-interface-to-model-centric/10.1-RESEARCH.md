# Phase 10.1: Refactor Query Interface to Model-Centric - Research

**Researched:** 2026-02-17
**Domain:** Python ORM query interface design, descriptor protocols, type safety
**Confidence:** HIGH (codebase fully analyzed, design aligned with established industry patterns)

## Summary

Phase 10.1 refactors Cubano's query API from a procedural builder (`Query()`) to a model-centric pattern (`Users.query()`). This research investigates the technical foundation already in place, validates the design decisions against established patterns, and identifies implementation options for the critical namespace collision problem.

The existing codebase already supports the core building blocks: frozen immutable `Query` dataclass with fluent chaining, Field descriptors with `__set_name__` validation, SemanticView metaclass enforcing immutability, operator overloading for filters (`&`, `|`), and a Result object with flexible row access. The refactor requires minimal new infrastructure—primarily adding a `.query()` class method to SemanticView and resolving how field attribute access coexists with method names (`query`, `metrics`, `dimensions`).

**Primary recommendation:** Use reserved method names pattern with careful documentation. Namespace collision is rare in practice (Looker, Metabase, Cube.dev all use reserved names), and Django ORM validates this approach. If sophisticated descriptor chaining is desired later, the infrastructure is in place to upgrade—but the simpler approach has no type-checking penalties and better IDE behavior.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Query Construction API:**
- Entry Point: Users start queries via model class method: `users.query()`
- Field Selection: Users select fields using model attributes: `query.metrics(Users.revenue, Users.users_count)`
  - Provides IDE autocomplete, catches typos at dev time, enforces type safety
  - Attribute access discovered via IDE hints first (primary discovery method)
- Filtering: Use operator overloads for WHERE conditions: `.where(Users.country == 'US')`
  - Simple boolean logic over Q-object patterns; more Pythonic
  - Support OR composition via operator: `(Users.country == 'US') | (Users.country == 'CA')`

**Result Handling & Execution:**
- Execution Model: Eager execution: `.execute()` returns results immediately
- Result Format: Custom Result object (typed, with helper methods)
  - Preserves row access pattern already implemented in codebase
  - Result object allows `.to_pandas()`, aggregation helpers, etc. as extensions
- Row Access: Preserve existing row access implementation (already working)

**Model Introspection:**
- Discovery Methods: Primary: IDE autocomplete; Fallback: Explicit `.metrics()` and `.dimensions()` methods
- Field Introspection Methods: `Users.metrics()` — returns List[Field]; `Users.dimensions()` — returns List[Field]
- Field Objects: Return rich Field objects with name, type, aggregation type, docstring, warehouse metadata

### Claude's Discretion (Research and Recommend)

- Exact descriptor / `__getattr__` implementation for graceful fallbacks
- Row serialization format and performance optimizations
- Helper methods on Result object (`.summary()`, `.pivot()`, etc.)
- IDE stub generation for autocomplete enhancement

### Deferred Ideas (OUT OF SCOPE)

**Namespace collision handling** — How to handle conflicts between built-in methods (query, metrics, dimensions) and user-defined field names?

Options to research:
1. Descriptors + `__getattr__` — Pythonic but complex, type-checking challenges, IDE confusion
2. Reserved method names — Simpler, clearer (document: "don't name a field 'query'"), but less flexible
3. `_fields` dict approach — Explicit and safe, but less pretty

This decision should be researched and recommended by the planning phase; deferred from context gathering.

## Architecture Patterns

### Pattern 1: Model-Centric Entry Point

**What:** Add a `@classmethod` to SemanticView that returns a Query instance bound to that model.

**When to use:** This is the REQUIRED primary entry point for users. All query construction should start here.

**Current codebase foundation:**
- SemanticView uses `__init_subclass__` to collect fields into `_fields` (MappingProxyType)
- SemanticViewMeta enforces immutability on the class
- Both mechanisms are in place and working

**Implementation pattern (from analyzed codebase):**

```python
# src/cubano/models.py (addition to SemanticView class)

@classmethod
def query(cls, using: str | None = None) -> Query:
    """
    Create a query for this semantic view.

    Args:
        using: Optional engine name (defaults to 'default')

    Returns:
        Query instance bound to this model's view
    """
    q = Query(_using=using)
    # Store reference to owning model for validation later
    q._model = cls
    return q
```

**Type safety benefit:** The returned Query "remembers" its model context. Field selection methods can validate that only fields from this model are used.

---

### Pattern 2: Field Introspection with Rich Objects

**What:** `.metrics()` and `.dimensions()` as class methods returning List[Field], not list of strings.

**When to use:** When user wants to explore available fields (REPL-friendly, IDE-discoverable).

**Current codebase foundation:**
- Field base class already stores `name` and `owner` (set by `__set_name__`)
- Metric, Dimension, Fact subclasses inherit all descriptor behavior
- `_fields` is a MappingProxyType preserving immutability

**Implementation pattern (from analyzed codebase):**

```python
# src/cubano/models.py (additions to SemanticView class)

@classmethod
def metrics(cls) -> list[Metric]:
    """
    Get all metric fields for this model.

    Returns:
        List of Metric descriptors with full metadata
    """
    return [f for f in cls._fields.values() if isinstance(f, Metric)]

@classmethod
def dimensions(cls) -> list[Dimension]:
    """
    Get all dimension fields for this model.

    Returns:
        List of Dimension descriptors with full metadata
    """
    return [f for f in cls._fields.values() if isinstance(f, Dimension)]
```

**Type safety:** `List[Metric]` vs `List[Dimension]` ensures IDE and type checker understand field roles.

**REPL usage:**
```python
>>> Users.metrics()
[<Metric object 'revenue'>, <Metric object 'users_count'>]
>>> Users.metrics()[0].name
'revenue'
```

### Pattern 3: Operator Overloading for Filters

**What:** Use Python's comparison operators (`==`, `>`, `<`, etc.) on Field objects to build filter expressions naturally.

**When to use:** Always, for WHERE clauses. This is LOCKED as the primary filter API.

**Current codebase foundation:**
- Q object already supports `&` (AND), `|` (OR), `~` (NOT)
- Field class uses descriptor protocol; Field instances are the primary user interface
- filters.py demonstrates composable boolean logic

**Implementation pattern (from codebase and decision):**

```python
# Usage pattern (from CONTEXT.md):
query.where(Users.country == 'US')
query.where((Users.country == 'US') | (Users.country == 'CA'))

# This requires Field.__eq__, Field.__gt__, etc.
# NOT currently in codebase — must be added in this phase
```

**Current gap:** Field class has `.asc()` and `.desc()` but no comparison operators. Need to add:
- `Field.__eq__(value)` → returns a Q object or filter expression
- `Field.__gt__`, `__lt__`, `__ge__`, `__le__` → similar
- These methods return Q-compatible objects that can compose with `&`, `|`, `~`

**Design choice needed during planning:** Should Field operators return Q objects directly, or intermediate "FieldCondition" objects? Query builder in phase 02 shows Q is the established pattern, so direct Q return is likely correct.

### Pattern 4: Descriptor-Based Field Access with Namespace Management

**What:** Decide how model attributes (field names) coexist with method names (`query`, `metrics`, `dimensions`).

**When to use:** This is critical for ANY model definition. Three options identified in deferred ideas; research findings below.

**Industry precedent (from MODEL_QUERY_INTERFACE_RESEARCH.md and codebase analysis):**

| Layer | Pattern | Method Names | Field Name Limits |
|-------|---------|--------------|-------------------|
| Django ORM | Reserved names | `.filter()`, `.aggregate()`, `.values()` | Documents: "don't use these as field names" |
| Looker LookML | Reserved names | `explore`, `view`, `dimension`, `measure` | Schema validator rejects reserved names |
| Metabase | Reserved names | `query`, `metric`, `filter`, `model` | JSON schema doesn't allow reserved |
| Cube.dev | Reserved names | `cube`, `dimension`, `measure` | Schema validation on model load |
| SQLAlchemy ORM | Reserved names + getattr | `.filter()`, `.select()` | `__getattr__` fallback, but reserved list is substantial |

**Option A: Reserved Method Names (RECOMMENDED)**
```python
# Field names that cannot be used:
reserved = {'query', 'metrics', 'dimensions', 'execute', 'to_sql', 'using', 'where', 'order_by', 'limit'}

# Users cannot define:
class Users(SemanticView, view='users'):
    query = Metric()  # ERROR: reserved name

# Validation in Field.__set_name__:
if name in reserved:
    raise ValueError(f"Field name '{name}' is reserved...")
```

**Pros:**
- No descriptor complexity; Field.__get__ works as-is
- IDE autocomplete works perfectly (methods show, fields show separately)
- Type checking has no special cases
- Easy to validate at class definition time
- All semantic layers use this pattern — proven approach

**Cons:**
- Limits field naming (but extremely rare in practice — warehouse column names almost never conflict)
- Requires documentation and clear error messages

**Option B: Descriptors + __getattr__ (COMPLEX)**
```python
# SemanticView.__getattr__ to resolve field access when attribute not found
# Field.__get__ returns self for class access
# __getattr__ only called if normal lookup fails

class SemanticView:
    def __getattr__(cls, name: str):
        if name in cls._fields:
            return cls._fields[name]
        raise AttributeError(...)
```

**Pros:**
- Technically allows any field name (even 'query')
- No validation needed at class definition time

**Cons:**
- `__getattr__` in metaclass (not instance) is non-standard and fragile
- IDE autocomplete sees method `query` first, then field `query` — confusing UX
- Type checkers (basedpyright strict) struggle with `__getattr__` resolution
- SQLAlchemy avoids this complexity; Django forbids it
- Harder to debug when field access is ambiguous

**Option C: Explicit _fields Dictionary (VERBOSE)**
```python
# Users access fields via Users._fields['revenue'] or Users['revenue']
class Users(SemanticView, view='users'):
    revenue = Metric()
    users_count = Metric()

# Usage:
Users._fields['revenue']  # or via custom __getitem__
```

**Pros:**
- Completely unambiguous; no collisions possible

**Cons:**
- Very ugly API (`Users._fields['revenue']` vs `Users.revenue`)
- Contradicts LOCKED decision: "Attribute access discovered via IDE hints first"
- Not how any semantic layer works

**Recommendation for planning phase:** Option A (Reserved Names). The existing Field class is already designed with this assumption (validates field names in `__set_name__`, line 80 in fields.py). The current reserved set in fields.py is: `{"keys", "values", "items", "get", "pop", "update", "clear"}` — designed to avoid dict method conflicts on Row objects. Expand this to include query builder methods when `.query()`, `.metrics()`, etc. are added.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Operator overloading for filters | Custom filter DSL | Python's `&`, `|`, `~` operators on Field | Python operators are familiar, compose well, already implemented in Q object |
| Field metadata storage | Global registry for field metadata | Descriptor protocol + MappingProxyType | Already in use; descriptor's `__set_name__` captures metadata at class definition time; metaclass enforces immutability |
| Query immutability | Custom copy logic | dataclass(frozen=True) | Already in use; dataclass.replace() handles immutable updates elegantly |
| Field validation | Custom validation in __init__ | Descriptor.__set_name__ validation | One-time validation at class definition time catches errors immediately; per-instance validation is redundant |
| Result row access | Custom Row class | Descriptor + object.__getattr__ + dict keys | Row class already exists and works; supports both attribute and dict-style access |
| Engine lookup | Hard-coded engine references | Registry pattern (singleton dict) | Already in place; lazy resolution allows engines to be registered after models are defined |

**Key insight:** The Cubano codebase is built on solid architectural choices that are proven in production systems. Do not refactor core infrastructure during this phase. Focus on the API-level changes: adding `.query()` entry point, field introspection methods, and operator overloads for filters.

## Common Pitfalls

### Pitfall 1: Breaking Type Safety with Flexible Query Binding

**What goes wrong:** If `Query.query()` entry point stores model reference but Query methods (`metrics()`, `dimensions()`) don't validate fields belong to the model, users can mix fields from different models in one query. The query builder will happily accept `Query().metrics(Users.revenue).metrics(Orders.total)` even though it's invalid.

**Why it happens:** Eager to preserve flexibility; validation seems like a constraint.

**How to avoid:** When adding `.query()` class method to SemanticView, enforce model consistency in Query methods. One approach:
```python
# In Query class
def metrics(self, *fields: Any) -> Query:
    # Check: if _model is set, all fields must belong to _model
    if hasattr(self, '_model') and self._model:
        for f in fields:
            if f.owner != self._model:
                raise TypeError(f"Cannot mix fields from different models...")
    # ... rest of validation
```

**Warning signs:** Users report "weird SQL generated" or "query runs but returns wrong results" — often caused by silently mixing fields.

---

### Pitfall 2: IDE Autocomplete Chaos from Namespace Collisions

**What goes wrong:** If `Users.query` (attribute) and `Users.query()` (method) both exist, IDE suggestions show both, user types `Users.query()` and doesn't know which one they're calling. Autocomplete menu is cluttered; type hints are ambiguous.

**Why it happens:** Trying to support "any field name" by using `__getattr__` or similar fallback. Each fallback adds uncertainty.

**How to avoid:** Accept reserved method names. Document them clearly in SemanticView docstring. Provide helpful error when user tries to define a reserved field:
```python
class Users(SemanticView, view='users'):
    query = Metric()  # Raises ValueError with clear message

# Error message:
# Field name 'query' is reserved. Reserved names: query, metrics, dimensions, execute, to_sql, using, where, order_by, limit
# Did you mean: 'user_query'? Or map the warehouse column: query = Metric(source='query')
```

**Warning signs:** Type checker reports "ambiguous field access" or IDE shows two definitions for same name.

---

### Pitfall 3: Forgetting Lazy Engine Resolution

**What goes wrong:** Models are defined and `.query()` is called before any engine is registered. Code throws "engine 'default' not registered" at query time, not at model definition time.

**Why it happens:** Validation at class definition time (eager binding) seems "safer," but it prevents flexible deployment patterns where engines are registered in a separate configuration module.

**How to avoid:** Query execution (`.execute()`, `.to_sql()`) should resolve engines lazily, not `.query()` itself. Current codebase already does this — `.fetch()` calls `get_engine(self._using)` at execution time. Preserve this pattern.

**Warning signs:** Code that worked in one environment breaks in another because engine registration order changed.

---

### Pitfall 4: Result Type Ambiguity (List[Row] vs Result Object)

**What goes wrong:** LOCKED decision says "Custom Result object," but current codebase returns `List[Row]`. If phase 10.1 changes execution to return a Result wrapper object (`.to_pandas()` helper, etc.), existing code breaks or needs migration.

**Why it happens:** Phase 10.1 wants "eager execution" and "Result object with helper methods," but query.py currently returns `list[Any]` from `.fetch()`.

**How to avoid:** Plan carefully:
1. Keep `.fetch()` returning `List[Row]` for now (backward compatible)
2. Add a new `.execute()` method that returns a Result object with helpers
3. Result object is a thin wrapper: `class Result: def __init__(self, rows): self.rows = rows`
4. Result has helper methods: `to_pandas()`, `summary()`, etc.

Alternatively, if planning phase decides to break to Result everywhere, do it consistently across both `.fetch()` and `.execute()`.

**Warning signs:** Tests pass but production code breaks; helper methods don't exist when expected.

---

### Pitfall 5: Forgetting Docstring Preservation

**What goes wrong:** LOCKED decision says "Preserve user's own docstring on model class (don't inject field lists)." If phase 10.1 adds a `.metrics()` method to SemanticView, it must not override or shadow user-written docstrings.

**Why it happens:** Trying to be "helpful" by auto-generating model documentation.

**How to avoid:**
1. `.metrics()` and `.dimensions()` are read-only class methods with their own docstrings
2. The model class docstring is untouched by the framework
3. Documentation tools can call `.metrics()` to generate field lists if desired, but this is not automatic

```python
class Users(SemanticView, view='users'):
    """
    Users semantic view with revenue and count metrics.

    This docstring is written by the user and never modified.
    """
    revenue = Metric()
    users_count = Metric()

# DO NOT inject:
# "Available metrics: revenue, users_count" into the docstring
```

**Warning signs:** User's docstring is mysteriously different after running code; documentation generation is unpredictable.

### Example 2: Model-Centric Query Entry Point

```python

from cubano import SemanticView, Metric, Dimension, Q

class Users(SemanticView, view='users'):
    revenue = Metric()
    users_count = Metric()

results = (Users.query()

from cubano import Query
results = (Query()

    .filter(Q(country='US'))
    .order_by(Users.revenue.desc())

# Note: .where() is new (LOCKED as primary); .filter() is old (internal)
# Note: .execute() is new (LOCKED, returns Result); .fetch() is old (returns List[Row])
```

**Why this pattern:** Matches Django ORM (`Model.objects.filter()`) and industry semantic layers (Looker, Metabase, Boring Semantic Layer). Model-centric entry makes it impossible to accidentally mix fields from different views.

### Example 3: Operator Overloading for Filters

```python
# Source: Proposed extension of Field class (fields.py)

from cubano import SemanticView, Dimension, Metric, Q

class Users(SemanticView, view='users'):
    country = Dimension()
    revenue = Metric()

# Phase 10.1 operator overloading (LOCKED pattern):
query = (Users.query()
    .where(Users.country == 'US')
    .where((Users.country == 'CA') | (Users.country == 'MX'))
    .where(Users.revenue > 10000)

# Under the hood:
# Users.country == 'US'  calls Field.__eq__('US') -> returns Q object
# (Q(...) | Q(...))  calls Q.__or__ -> returns combined Q
# .where(q)  is same as .filter(q) but LOCKED as primary method name
```

**Why this pattern:** Python operators are familiar and Pythonic. Compose with `&` (AND), `|` (OR), `~` (NOT). Q object already implements this, just need Field operators to return Q objects.

## State of the Art

This phase refactors an interface designed in phases 01-04 based on research that shows semantic layers universally converge on model-centric patterns. The research that informed current locking decisions (MODEL_QUERY_INTERFACE_RESEARCH.md) surveyed:

| System | Pattern | Precedent |
|--------|---------|-----------|
| Cube.dev | View-centric (internal cube joins) | High-level analytics; joins pre-defined |
| Looker LookML | Explore-centric (LookML defines joins) | BI tool; joins in schema, not query |
| Metabase | Model-centric | Analytics tool; one model per question |
| Django ORM | Model-centric (`Model.objects`) | Web framework; foundational Python pattern |
| Boring Semantic Layer | Model-centric (`model.query()`) | Modern Python semantic layer |
| Snowflake Semantic Views | Single-view per query | Warehouse-native; `AGG()` only works within one semantic view |
| Databricks Metric Views | Single metric-view per query | Warehouse-native; metric joins to regular tables only |

**Phase 10.1 represents the final step in this evolution:** moving Cubano's API to match what all major systems have settled on. The current codebase already implements the building blocks correctly; this phase connects them.

**No deprecated features:** Nothing from phases 01-04 is being removed in the traditional sense. The `Query()` constructor remains (for internal use or advanced scenarios). The `.filter()` method remains (becomes `.where()` at user level but `.filter()` internally). The `.fetch()` method remains (becomes `.execute()` at user level). This is additive, not destructive.

## Open Questions

### 1. Reserved Method Names vs Flexible Namespacing

**What we know:**
- LOCKED decision says attribute access is primary discovery method
- Industry standard (Django, Looker, Metabase, Cube.dev) uses reserved names
- Current codebase already validates field names in `__set_name__`
- Option A (reserved names) has zero type-checking penalties; Option B (descriptors) has significant challenges

**What's unclear:**
- How many real-world use cases have warehouse columns named `query`, `metrics`, `dimensions`? (Hypothesis: almost none — SQL reserved words are avoided in column naming)
- Would users accept reserved names documentation, or is this a deal-breaker?
- Should error message suggest alternate names, or require manual rename?

**Recommendation:** Plan for reserved names (Option A). If future feedback shows this is a problem, infrastructure is in place to upgrade to descriptors, but the simpler approach is recommended now.

---

### 2. Result Object Scope and Helper Methods

**What we know:**
- LOCKED: Custom Result object returned by `.execute()`
- LOCKED: Preserve row access pattern from codebase (Row object with attribute and dict-style access)
- Claude's Discretion: Helper methods (`.summary()`, `.pivot()`, etc.)

**What's unclear:**
- How much Row compatibility is needed? If Result wraps List[Row], does `for row in result` work, or must users access `.rows`?
- What helper methods should Result have? `.to_pandas()` is mentioned, but `.summary()` and `.pivot()` are not defined.
- Should Result be typed `Result[Row]`, or a concrete class?
- Does `.execute()` replace `.fetch()` or coexist?

**Recommendation:** Planning phase should spec this carefully. Suggest:
- Result is a simple class: `class Result: def __init__(self, rows: list[Row]): self.rows = rows`
- Implement `__iter__()` and `__getitem__()` to proxy to `.rows` for backward compat
- Start with `.to_pandas()` (straightforward); defer `.summary()` and `.pivot()` to phase 10.2 or later
- `.execute()` is the new primary method; `.fetch()` becomes internal or deprecated

---

### 3. Filter Method Naming (`.filter()` vs `.where()`)

**What we know:**
- LOCKED: Use `.where()` for WHERE conditions (primary user-facing API)
- Current codebase: `.filter()` is the implemented method
- Query.filter() internally accepts Q objects and combines with AND

**What's unclear:**
- Should `.filter()` be removed or kept as internal/alias?
- Django uses `.filter()`, SQL uses WHERE — which takes precedence in naming?
- If both exist, does that create confusion?

**Recommendation:** Planning should decide:
1. Add `.where()` as public method; deprecate `.filter()` (or keep as alias)
2. Or: rename `.filter()` to `.where()` at public API level, keep internal name as needed

Pattern from codebase: already uses `.filter()` internally for Q objects. Simplest migration: add `.where()` as alias, update documentation and examples.
