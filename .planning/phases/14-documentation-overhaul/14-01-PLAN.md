  <action>
    Create `CLAUDE.md` in the project root with documentation quality gate rules. This file instructs Claude (and any future AI contributors) on project conventions.

    Content must include these sections:

    **1. Project overview** (2-3 lines): Cubano is a Pythonic ORM for querying data warehouse semantic views (Snowflake and Databricks). Python >=3.11, built with uv.

    **2. Quality gates** (from MEMORY.md):
    - Typecheck: `uv run basedpyright`
    - Lint: `uv run ruff check`
    - Format: `uv run ruff format --check`
    - Tests: `uv run --extra dev pytest`
    - Docs build: `uv run mkdocs build --strict`

    **3. Documentation standards** -- this is the critical new section:

    **Mandatory skills:** When writing or modifying documentation, you MUST use:
    - `diataxis-documentation` skill -- for content structure and classification (load SKILL.md, then the appropriate references/ file for the doc type)
    - `humanizer` skill -- for tone review on all new or substantially rewritten prose

    **When to apply skills:**
    - New pages: full diataxis classification + humanizer pass (mandatory)
    - Major rewrites (>50% of page changed): full diataxis + humanizer pass (mandatory)
    - Minor fixes (typos, version numbers, small corrections): skills not required
    - API surface changes: update corresponding docs (mandatory), but full rewrite not required

    **Content type standards:**
    - Tutorials (docs/src/tutorials/): Runnable code with imports and expected output shown. Learning-oriented, guided step-by-step.
    - How-to guides (docs/src/how-to/): Illustrative snippets showing key concepts. Goal-oriented, reader supplies setup. Use MkDocs tabbed content (`=== "Snowflake"` / `=== "Databricks"`) for SQL dialect examples.
    - Reference (docs/src/reference/): Auto-generated via mkdocstrings. Do not hand-write API docs.
    - Explanation (docs/src/explanation/): Background concepts, no step-by-step instructions. Link to tutorials/how-tos for action items.

    **Writing voice:**
    - Audience: Data/analytics engineers comfortable with SQL and Python
    - Tone: Warm but efficient (like FastAPI/Stripe docs)
    - Perspective: Second person ("you")
    - Self-contained pages with "See also" links at bottom for cross-referencing

    **4. Style** (from MEMORY.md):
    - Line length: 100 chars
    - Docstrings: opening/closing `"""` on own lines for multi-line
    - Avoid `# type: ignore`

    Keep CLAUDE.md concise and scannable. Use headers and bullet points. Target ~60-80 lines total. Do NOT include project history or planning details -- this is a conventions file for contributors.
  </action>
